# Mycelium v2.0 — consolidated plan

**Status:** planning. v1.x is engineering-complete and shipped (`v1.2.0`, 2026-06-14 —
Production Readiness Gap closed: WS1 RBAC, WS2 audit, WS3 crown-jewel, WS4 OIDC SSO,
WS5 hot cert rotation). This document is the **execution layer** for the next major
version.

**Relationship to [`ROADMAP.md`](../../ROADMAP.md):** this plan does *not* restate
milestone designs. [`ROADMAP.md` §v2.0 Milestones](../../ROADMAP.md) remains the
canonical home for each milestone's detailed write-up (problem, mechanism, compliance
with Core Principles, trigger). This plan adds the three things the scattered milestone
notes don't carry: **(1)** a workstream grouping, **(2)** a dependency / critical-path
graph, and **(3)** a completeness matrix + definition-of-done so that "v2, once
delivered, covers all areas discussed." Read this for *strategy and sequencing*; follow
a link into ROADMAP for *the design of any one item*.

> ROADMAP milestones are cited below as **M1…M16** (their numbered position in
> `ROADMAP.md` §v2.0 Milestones). Two cross-cutting design notes that live *after* M16 in
> ROADMAP are cited as **[Quilt-DD]** (NANDA Registry-Quilt deep dive — technique-transfer
> candidates) and **[Schema-Evo]** (schema-registry evolution).

## Posture — v2 is a demand-driven menu, not a forced march

Every milestone in ROADMAP carries a **Trigger to start**, and that stays load-bearing:
v2 has no single ship date and no obligation to build all of it. The substrate is shipped
and production-ready at v1.x; each v2 item is built **when a real workload pulls it**.

The job of *this* plan is therefore **coherence and completeness**, not scheduling:
- ensure the items group into coherent workstreams that can be reasoned about and shipped
  independently;
- make the dependencies explicit so nothing is started before its precursor;
- guarantee — via the completeness matrix — that every theme raised in design discussion
  maps to a workstream, so "v2 is done" is a checkable claim rather than a vibe.

This also means **Core Principle 1 (no coordinator)** is the gate every workstream passes
through. Where a milestone touches provisioning, supervision, placement, or admission, the
coordinator-free shape is stated inline in ROADMAP and must not be eroded for convenience.

## Workstreams

### WS-A — Crate & API structure
*Theme: embeddability, smaller dep trees, dissolve the `TaskCtx` God Object.*

| Item | ROADMAP | Note |
|---|---|---|
| Workspace split → `mycelium-core` (Layers I+II) | M1 | The structural keystone; absorbs the Layer I/II entanglement item by drawing the crate boundary *around* it at the II↔III seam. `CoreCtx` cleaves the 22-field `TaskCtx`. |
| `#[cfg(feature = "consensus")]` gate | M2 | Lets minimal embeds drop the Paxos machinery. |
| Owned standalone handles | M3 | `KvHandle` / `MeshHandle` / `CapabilitiesHandle` without a live `GossipAgent` borrow. |

**Done when:** a pure Layer-I+II embed compiles with a materially smaller dep tree, with no
consensus/capabilities/gateway/tls, and handles are passable across crate boundaries.

---

### WS-B — Scale & transport
*Theme: break the O(N²) connection ceiling; bound bytes-on-wire.*

| Item | ROADMAP | Note |
|---|---|---|
| Partial-mesh gossip (bounded fan-out) | M4 | Lighter precursor; largely subsumed by M5. |
| Hybrid TCP/UDP transport (SWIM-style) | M5 | The structural fix for the iptables FORWARD-chain ceiling. NANDA's Quilt independently chose SWIM — convergent validation. |
| Wire-codec succession (bincode → hand-rolled fixed-layout) | M11 | Rides the next `WIRE_VERSION` bump (v12). |
| **Merkle-tree anti-entropy** | [Quilt-DD] #1 | *Net-new from the NANDA dive.* Current sweep ships the full `key_timestamps` index every probe (O(N) on the wire); Merkle reconciliation descends only divergent subtrees. **Rides M11's v12 bump** (changes `StateRequest` shape) and **sequences with M5** (both touch the anti-entropy path). |

**Done when:** a >500-node cluster forms without iptables saturation, per-node connection
memory is O(1) for heartbeats, and the anti-entropy sweep tail (`make test-scale-entries`)
is bounded by divergence, not store size.

### WS-C — Self-managing metabolism
*Theme: the cluster tunes itself to its own size and load.*

| Item | ROADMAP | Note |
|---|---|---|
| Startup-time auto-derivation | M8 | Closed-form formulas from N; `None`/"auto" sentinels in `GossipConfig`. |
| Hot-reloadable tuning subset | M9 | `Arc<AtomicU32>` params + a `ClusterTuner` app-layer agent over `sys/config/`. |
| Full live reconfiguration (coordinated fence) | M10 | Consensus-fenced restart of timing params; the heavy case. |
| Cluster-wide distributed rate-limiting | M7 | Gossip per-sender observations + consensus backpressure. |

**Done when:** an operator can deploy a cluster of any size without hand-setting tuning env
vars, and the cluster adapts to elastic scaling without manual reconfiguration.

---

### WS-D — Security & trust hardening
*Theme: deepen v1.x compliance from gateway-level to gossip-level + federation-grade revocation.*

| Item | ROADMAP | Note |
|---|---|---|
| Full gossip-level capability authorization | M6 | Write-ACL enforced at every forwarding hop, not just the gateway; cluster role policy via consensus. |
| **CT-style transparency log + inclusion proofs** | [Quilt-DD] #2 | *Net-new from the NANDA dive; the most federation-aligned item.* Per-domain, gossip-replicated append-only log of key/rotation/revocation events. Delivers "sub-second revocation," closes the **WS5** "compromise needs explicit revocation" caveat, and extends the per-node **WS2** audit chain — **without** a central CT operator (Core Principle 1). |

**Done when:** capability writes are authz-enforced at the gossip layer, and a compromised
key can be revoked cluster-wide with client-checkable proof, not just rotated.

---

### WS-E — Code mobility & autonomic provisioning
*Theme: dynamic capability install without a coordinator — the "better BRAIN-IoT" story.*

| Item | ROADMAP | Note |
|---|---|---|
| WASM components (runtime code mobility) | M12 | The OSGi-bundle analogue Rust natively lacks: `mycelium-wasm-host` companion crate, content-addressed pull + verify + instantiate. **The install *mechanism*.** |
| OBR-style resolve-from-installable-catalog | M15 | **The selection step.** Install-time resolution is **one hop, not a constraint solver** (design contract): service/capability deps are runtime-mesh-resolved; only direct install-artifact deps resolve, bottoming out at host-satisfied WIT imports. Depends on M12. |
| Substrate-native supervision | M14 | Capability-presence invariants; emergent self-election + leased-consensus for singletons. **Pairs with M15:** a supervised role whose provider evaporated is just an unmet requirement the provisioner re-satisfies — restart and first-time provisioning collapse onto one resolve-and-pull path. |

**Context (BRAIN-IoT, corrected — see [[user-brainiot-coauthor]]):** BRAIN-IoT's autonomic
requirement→deploy loop was *already decentralized* (event bus + per-node BMS + OSGi
resolver). Mycelium's edge is **not** coordinator removal — it is **edge-triggered event →
level-triggered soft-state**: the requirement is evaporating `req/` state (reconcile-to-
desired), not a transient bus event, with demand / backpressure (opacity) / failure-recovery
(evaporation) / dedup (claim-by-sentinel) folded into one gossip consistency model.

**Done when:** a fleet provisions capabilities *by need* — nodes acquire what unmet
requirements call for, self-heal when a provider dies — with no operator naming artifacts and
no coordinator assigning placement.

### WS-F — Federation & interop
*Theme: make a Mycelium domain a sovereign patch in the agent-internet quilt.*

| Item | ROADMAP | Note |
|---|---|---|
| NANDA interop — AgentFacts emission (PULL, edge) | M16-A / [Quilt-DD] #4 | Self-signed AgentFacts (JSON-LD) at the gateway edge, a superset of the A2A AgentCard Mycelium already serves. Self-certified only (no issuer/TRS authority — Core Principle 1). |
| NANDA interop — CRDT AgentFacts update layer (PUSH, intra-domain) | M16-B | Per-field-signed (`SignedData`) AgentFacts gossiped intra-domain; LWW+HLC+anti-entropy *is* the "CRDT update protocol" NANDA's abstract names but doesn't deliver. |
| OR-Map CRDT for the capability/registry projection | [Quilt-DD] #3 | *Design note, not a rewrite.* Observed-remove semantics for `gcap/` where concurrent add/remove is the norm — evaluate, don't replace LWW+HLC wholesale. |
| Schema-registry evolution — runtime migration | [Schema-Evo] | Additive tolerance (verify it's already true on JSON paths) → mismatch detection (a tripwire) → **registered, gossip-distributed migrations** (the v2 aspiration; explicit, never silent coercion). Pairs with M16-B's evolvable AgentFacts. |

**Done when:** a domain can publish self-certified AgentFacts at its edge (opt-in, run-dark by
default), keep them concurrently updated intra-domain on the existing substrate, and survive
schema evolution via explicit migrations — discoverable across a NANDA-style quilt without
adopting any coordinator-shaped trust authority.

```
       M14 ──────────┘

       M16-B ◄┄ benefits from ┄ [Schema-Evo] registered migrations
       [Quilt-DD#3] OR-Map = design note on the cap projection

       M13 + tuple-space WAL discipline ──► mycelium-blackboard
```

**Three cross-workstream couplings to honour:**
1. **Merkle anti-entropy [Quilt-DD#1] rides WS-B's wire/transport work** (M11 v12 bump +
   M5's anti-entropy rework) — don't land it as a standalone wire break.
2. **WS-E is a strict chain:** M12 (mechanism) → M15 (selection) → M14 (recovery as
   re-provisioning). M15 and M14 share the resolve-and-pull path.
3. **WS-F's schema-migration [Schema-Evo] precedes / accompanies M16-B**, whose AgentFacts
   need field-level evolution.

## Completeness matrix — every discussed theme maps to a workstream

This is the "v2 covers all areas discussed" guarantee. No row may be orphaned.

The **Layer** column is the structural defense of "library, not platform" ([[feedback-deployment-framing]]):
every platform-like behaviour (self-tuning, supervision, provisioning, federation) must live in a
**companion crate** or an **app-layer agent** on the public API — *not* accrete into the substrate.
**S** = substrate (core mechanism); **C** = companion crate; **A** = app-layer agent. Anything that
would make `mycelium` itself a platform is a smell; the tuple-space proved the companion pattern.

| Theme raised in design discussion | Lands in | Item(s) | Layer |
|---|---|---|---|
| `TaskCtx` God Object / Layer I/II entanglement | WS-A | M1 | **S** (crate split) |
| Minimal embeds / smaller dep tree / no-std-friendliness | WS-A | M1, M2, M3 | **S** |
| 100-node iptables ceiling / O(N²) connections | WS-B | M4, M5 | **S** (transport) |
| Unmaintained bincode (RUSTSEC-2025-0141) | WS-B | M11 | **S** (wire codec) |
| Anti-entropy bytes-on-wire efficiency (NANDA Merkle) | WS-B | [Quilt-DD#1] | **S** (anti-entropy) |
| Cluster self-tuning / elastic scaling | WS-C | M8, M9, M10 | **S** sentinels + config sampling; **A** `ClusterTuner` |
| Intra-cluster abuse / rate-limiting | WS-C | M7 | **S** local budget; observations are KV |
| Gossip-level (not just gateway) capability authz | WS-D | M6 | **S** resolve-time gate + write-time tripwire |
| Sub-second revocation / WS5 compromise caveat (NANDA CT log) | WS-D | [Quilt-DD#2] | **S** (verify-on-read) |

## Philosophy compliance — guardrails (validated 2026-06-15)

The plan was audited against the six Core Principles (no coordinator; consistency-as-service;
layer discipline; detection-not-prevention; emergent/threshold; composability) and the
Holland/Hayek grounding. WS-A, WS-E, WS-F, WS-G hold cleanly. Three items carried
prevention- or authority-shaped wording (mostly inherited from pre-v1.x-hardening ROADMAP
text) and have been **re-aligned in ROADMAP**; their compliant shape is recorded here as a
binding guardrail so it is not re-litigated at build time. Plus one meta-guardrail and one
watch-item.

1. **M6 (gossip capability authz) — enforce at resolve/serve, detect at write.** *No Layer I
   write guard, no forwarding-hop block* — that would teach the gossip layer a higher-layer law
   (the inversion the commit-conflict and `sys/` tripwires exist to avoid). The resolver gate on
   signed-role ACLs is the enforcement point (consumers route around unauthorized providers);
   write-time is detection only (tripwire + counter). The advertisement still propagates per LWW.

2. **M7 (rate-limiting) — shared observation, local decision.** Gossiped `sys/rate/` data is
   *evidence*, not a verdict. Each node independently tightens its own inbound budget on local
   evidence (Principle 5); disconnection is a node-local self-defense choice. **No cluster-wide
   consensus eviction verdict** — that is a coordinator enforcing a behavioural judgment.

3. **M9 (`ClusterTuner`) — advisor, not config coordinator.** A node applies a `sys/config/`
   recommendation *only if its local policy accepts it*; application is never mandatory. The tuner
   advises, the node decides (Principle 1) — otherwise it acquires standing agency over cluster
   config.

**Meta-guardrail — library, not platform.** v2's self-tuning / supervision / provisioning /
federation is exactly the feature set that pulled BRAIN-IoT into platform-hood. The defense is the
**Layer column** above: every platform-like behaviour ships as a **companion crate or app-layer
agent on the public API** (the tuple-space proof), never as substrate. The substrate stays a
library; the platform is an emergent composition the operator assembles. An item drifting from
**C/A** into **S** is a design smell to challenge.

**Watch-item — OR-Map CRDT ([Quilt-DD#3]).** Must be an *application-level interpretation layered
on LWW entries*, not a second substrate merge rule — otherwise it fractures "single KV substrate,
LWW is the law." The "design note, not a rewrite" framing is load-bearing.

**Net:** with these guardrails recorded, there are **no open philosophy conflicts** in the plan —
the three tensions convert from unreconciled wording into binding design contracts. They are
*addressed* at the design/contract level; they are *honoured* when each milestone is implemented
to the shape stated here. (This section, plus the per-milestone *compliant shape* in ROADMAP, is
the standing check.)

## Definition of done for v2.0

v2.0 is **delivered** when, for each workstream, *either* its **Done when** clause is met,
*or* the workstream is explicitly deferred-with-reason (its triggers never fired) and that
deferral is recorded here. The completeness matrix above must have **no orphaned theme** at
release.

This is deliberately an *acceptance gate*, not a schedule. Because v2 is demand-driven, a
legitimate v2.0 release may ship some workstreams and consciously defer others — what it may
**not** do is leave a discussed theme unaccounted for (neither built nor explicitly deferred).

**Per-workstream status (update as items land):**

- [ ] **WS-A** Crate & API structure (M1, M2, M3)
- [ ] **WS-B** Scale & transport (M4, M5, M11, Merkle)
- [ ] **WS-C** Self-managing metabolism (M7, M8, M9, M10)
- [ ] **WS-D** Security & trust hardening (M6, CT log)
- [ ] **WS-E** Code mobility & autonomic provisioning (M12, M15, M14)
- [ ] **WS-F** Federation & interop (M16-A, M16-B, OR-Map, schema migrations)
- [ ] **WS-G** Coordination patterns (M13, blackboard)
