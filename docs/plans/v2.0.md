Production Readiness Gap closed: WS1 RBAC, WS2 audit, WS3 crown-jewel, WS4 OIDC SSO,
WS5 hot cert rotation). **WS-A (crate & API structure: M1 workspace split, M2 consensus

**Relationship to [`ROADMAP.md`](../../ROADMAP.md):** this plan does *not* restate
milestone designs. [`ROADMAP.md` §v2.0 Milestones](../../ROADMAP.md) remains the

graph, and **(3)** a completeness matrix + definition-of-done so that "v2, once

## Posture — v2 is a demand-driven menu, not a forced march

Every milestone in ROADMAP carries a **Trigger to start**, and that stays load-bearing:
v2 has no single ship date and no obligation to build all of it. The substrate is shipped
and production-ready at v1.x; each v2 item is built **when a real workload pulls it**.

The job of *this* plan is therefore **coherence and completeness**, not scheduling:
- ensure the items group into coherent workstreams that can be reasoned about and shipped
  independently;
- make the dependencies explicit so nothing is started before its precursor;
- guarantee — via the completeness matrix — that every theme raised in design discussion
  maps to a workstream, so "v2 is done" is a checkable claim rather than a vibe.

This also means **Core Principle 1 (no coordinator)** is the gate every workstream passes
through. Where a milestone touches provisioning, supervision, placement, or admission, the
coordinator-free shape is stated inline in ROADMAP and must not be eroded for convenience.

*Captured before WS-E opens. No milestone-scope changes — the M12→M15→M14 definitions hold —

##### E.1 — Logical model: the autonomic loop (level-triggered)

```
requirement declared          a provisioner AGENT (app-layer) observes demand
(req/{node}/{ns}/X)   ──►   demand_snapshot: want=N, provide=0   ──► self-elects
       ▲                                                                  │
       │ demander keeps                                  resolve req-filter against the
       │ refreshing req/                                 `installable` CATALOG  (M15)
       │ (evaporating soft-state)                                        │
       │                                          pull + verify + instantiate (M12)
       │                                          cap/{node}/X/loading: 0→100
       │                                                                 │
       └──────  cap/{node}/X live, gossips  ◄──  advertise_capability(X)
                demand relieved (want=N, provide≥1)
```

- **Where the requirement comes from — it is *declared*, never *computed* by the substrate**
  (the substrate has no agency to invent one — Core Principle 1). Three legitimate origins, all
  *above* Layer I: (1) a **runtime call dependency** — an agent needs to invoke a capability with
  no live provider; (2) a **group/role policy** — an unsatisfied `CapabilityGroupDef.requires`, or
  an M14 supervised-role invariant whose provider died; (3) an **explicit intent** — a human or a
  planning agent publishes "the fleet should be able to do X". This is the *management = intent +
  local reconcile* pattern: the requirement **is** the intent; resolve-and-pull **is** the reconcile.
- **Withdraw vs uninstall are distinct.** When a requirement evaporates (demander stopped
  refreshing) or providers exceed `max` (Track 2b), a governor **retracts the advertisement**
  (`cap/` tombstone — cooperative self-removal, never coordinator eviction). Deleting the *bytes*
  is a **separate, local resource-policy** decision (keep cached for cheap re-advertise, or evict
  under disk pressure). The substrate never forces an uninstall.
- **Restart = first-time provisioning (M14 ≡ M15 path).** A provider that dies has its `cap/`
  entry **evaporate** (refresh stops; `CapEntry::is_fresh`); the still-refreshed `req/` re-raises
  demand; the loop re-fires elsewhere. Self-heal needs no separate restart machinery.
- **Dedup is *soft*, by design.** Multiple eligible nodes may self-elect; the Track-2a
  probabilistic self-election + cooldown damps the thundering herd, and any over-provisioning
  self-corrects via withdraw (providers>max). The per-node `cap/{node}/{ns}/loading` marker
  prevents a node double-pulling and publishes progress. This is *threshold-over-agreement*
  (Principle 5), not a hard cross-node lock — a hard claim is deliberately **not** introduced.

##### E.2 — The host ⇄ component boundary (the most important M12 artifact)

A pulled WASM component runs **sandboxed, in-process** (wasmtime). It touches the node **only**
through a **WIT world** — the typed host/guest contract. *This boundary is the local API between
node and component.* It is **in-process FFI over the Component-Model canonical ABI, not a socket.**

- **Host *imports* (what the component may call)** = a **capability-scoped projection of the
  existing public handles** (`KvHandle`/`MeshHandle`/`ServiceHandle`/`CapabilitiesHandle`):
  scoped `kv` (confined to `cap/{me}/{ns}/*`), `mesh` emit/subscribe, `rpc` call, `caps`
  resolve/demand, `log`/`config`/`clock`. **Component *exports*** = the capability entry points
  the host calls on inbound invocation (`handle(request)->response`).
- **WIT imports = requires; WIT exports = provides** — directly realising M15's one-hop contract.
  A component imports the **mesh**, not other capabilities: a call to "skill B" goes
  `rpc.call(filter-for-B)` → **runtime-mesh-resolved**, never link-time-bound. That is *why* the
  install closure stays shallow (host-satisfied imports = depth 0; at most one artifact hop).
- **This boundary is the one place enforcement becomes *prevention*.** Everywhere in the
  substrate we do detection-not-prevention because Layer I must not police higher *layers*. The
  WASM guest is **not a layer — it is untrusted foreign code in the node's own process**, so the
  host *mediating every import* is legitimate prevention (reject out-of-namespace `kv.set`,
  unauthorised `mesh.emit`, disallowed egress via the existing `EgressPolicy`; attribute
  host-calls to the component identity → audit). This is **additive** to and does not weaken the
  substrate's detection posture (Principle 4 intact).
- **Two host/component flavours, one logical contract.** In-process WASM (M12: WIT world, canonical
  ABI, wasmtime sandbox) and out-of-process processes (**already shipped**: the gateway
  `/gateway/{kv,signal,rpc,…}` over localhost, scope-gated, used by the Python/TS SDKs) expose the
  *same* KV/mesh/rpc/caps surface over two transports. M12's WIT world is the in-process projection
  of what the gateway already projects over HTTP — **define the contract once, bind it twice.**
- **M12 owes, in priority order:** (1) the **WIT world** (host API) — the single most important
  design artifact; (2) the **host-side implementation** mapping each import to the scoped public
  handle + enforcing confinement; (3) the **invoke path** wiring an inbound capability call to the
  component export. Pull / verify / instantiate are mechanism *around* this.

##### E.3 — Operational model (made explicit): one loop, a spectrum — not a mode switch

There is **no deployment-mode flag**. Because a pre-bundled capability and a provisioned one are
**indistinguishable once advertised** (both are `cap/` entries; both gossip, evaporate, can be
withdrawn or supervised), "how a node got its capability" is invisible to the substrate. The three
modes are just points on one spectrum of *how much you pre-bundle vs. leave to provisioning*:

| Mode | What it is | What it needs |
|---|---|---|
| **1. Fixed / pre-bundled** | Nodes ship with all capabilities baked in; requirements met at startup; the loop never fires (dormant). | **Nothing new — works today.** This is the current product. |
| **2. Virgin / self-assembling** | Nodes start with only substrate + WASM host + a catalog source; inject a requirement → nodes resolve + pull + install + advertise. | M12 + M15 + a provisioner agent. |
| **3. Mixture (the realistic default)** | A pre-bundled baseline (critical/always-on caps) + on-demand provisioning for the long tail / bursty / optional caps. | Same as mode 2; mode 1 is the subset that is pre-bundled. |

Consequences worth stating: a **pre-bundled** capability whose process dies can *still* be
**re-provisioned by M14** if the artifact is in the catalog (restart ≡ provision); and Track 2b
**elastic sizing** applies to *any* `cap/` regardless of origin. So mode 3 is not a compromise —
it is the general case, with modes 1 and 2 as its endpoints. **Done-when** (above) is mode-2/3;
mode 1 is the always-available degenerate case.

##### E.4 — Build-on reuse (WS-E shrank; do not rebuild what exists)

1. **M15 provisioner + M14 supervision = `IntentReconciler` instances**, not new machinery — both
   are reconcile-to-desired loops; the Track-1 transport (`src/agent/intent.rs`: `FleetIntent` +
   `spawn_intent_reconciler` + `read_fresh_intent`) is exactly that shape. *Touch-point:* those
   items are `pub` but in the private `mod intent` — **promote to public API** for the
   `mycelium-wasm-host` crate / app-layer provisioner.
2. **M14 singleton election = the Track-2a engine** (`membership_governor.rs`: probabilistic
   self-election + drain + cooldown). "Which node restarts the dead singleton" ≡ "which node
   self-elects". Reuse/generalize; leased-consensus only for a genuinely inviolable singleton.
3. **Track 2b (provider/replica elastic sizing) is an explicit M15 deliverable** — a
   provisioner-driven elastic governor (provide when demand>providers & under `max`; withdraw when
   over `max`). Recorded here so the WS-C tail is not re-lost.
4. **Bulk-fetch client is a real M12 prerequisite (don't assume).** The content-addressed **serve**
   side is public (`ServiceHandle::bulk_serve`); the **pull/fetch client** (`BulkTransport::fetch`)
   is internal (bulk-signal handshake). M12 must surface a public "pull bytes by content address"
   path **or** use an external registry (Warg/OCI) for artifact bytes + Mycelium bulk for
   intra-cluster distribution.
5. **App-layer provisioner already satisfiable — no new substrate surface.**
   `CapabilitiesHandle::demand()` **and** `watch_demand()` are already public (the latter a reactive
   `watch::Receiver<DemandStatus>` — a ready-made trigger). The "library never auto-provisions"
   invariant (`demand.rs`) holds with the existing API.
6. **Hygiene invariants:** (a) WS-B — bincode retired (RUSTSEC); catalog/artifact metadata uses
   `serde_fixint` (KV-value path already does). (b) WS-A — M12 is its own companion crate;
   **wasmtime must not leak into `mycelium`/`mycelium-core`** (preserves core's ~48-dep win), same
   shape as `mycelium-tuple-space`.
7. **Operator visibility (optional)** — provisioning/supervision state can reuse the Track-3
   pattern (gateway routes + WS2 audit + per-node Prometheus gauges); not load-bearing for the engine.

##### E.5 — Conformance check (philosophy / architecture / implementation as of HEAD)

Verified against the 6 Core Principles, the crate architecture, and the current tree:

- **P1 (no coordinator / mechanism-not-agency):** ✅ provisioner is an app-layer agent; substrate
  contributes only primitives (`req/`, `demand_snapshot`, `installable`, bulk transport,
  `advertise_capability`). No node is *assigned* to provision — each self-elects on local
  knowledge. The host-boundary prevention is the node controlling *its own* guest, not a fleet
  coordinator. Confirmed in-tree: `demand.rs` "the library never auto-advertises".
- **P2 (consistency as a service, not foundation):** ✅ the loop is pure soft-state/LWW;
  consensus appears only optionally (M14 leased-consensus for a true singleton).
- **P3 (layer discipline):** ✅ `req`/`cap`/`installable`/`loading` are namespace conventions, not
  `apply_and_notify` write-guards; the host enforcement lives at the host/guest boundary (a
  companion crate), never in Layer I.
- **P4 (detection not prevention):** ✅ substrate stays detection-only; the *new* prevention is
  strictly at the untrusted-guest boundary — a different trust domain, additive, not a Layer-I law.
- **P5 (emergent / threshold over agreement):** ✅ self-election + soft dedup + withdraw-on-excess;
  no hard cross-node claim. Self-heal is evaporation-driven (`CapEntry::is_fresh`).
- **P6 (composability on the public API):** ✅ `mycelium-wasm-host` is a companion crate on the
  public API; the WIT world *is* the public-handle surface; out-of-process flavour already proven
  by the gateway + SDKs.
- **Architecture/impl facts confirmed at HEAD:** `demand()`/`watch_demand()` public
  (`capability_handle.rs`); `bulk_serve` public, `BulkTransport::fetch` internal (the §E.4.4 gap);
  `installable`/`loading` conventions documented (`lib.rs`); intent transport `pub` but in a
  private module (the §E.4.1 promotion). **One honest gap surfaced:** the roadmap's
  "dedup (claim-by-sentinel)" has no provisioning-level implementation today — but §E.1 resolves
  it as *intentionally soft* (self-election damping), so this is a clarification, not new work.

**Verdict:** the WS-E plan is coherent with the philosophy, consistent with the architecture, and
cohesive with the implementation as it stands. No foundations are undermined; WS-E builds on them.

#### WS-F pre-flight (design decisions, 2026-06-19 — captured before M16-A opens)

*M16's design is fully specified in ROADMAP §16; this pins the build decisions + the one
dependency surfaced. Sequencing: **M16-A (PULL/edge emission) first/standalone**, then M16-B
(intra-domain PUSH/CRDT) on top, then the OR-Map + schema-evolution design notes.*

- **F.1 — Companion crate `mycelium-agentfacts`, public API only.** Same composability proof as
  `mycelium-tuple-space` / `mycelium-wasm-host` — never core bloat. Run-dark by default: the crate
  is inert until an operator wires it in (the domain self-elects federation — Principle 1).
- **F.2 — Decouple from NANDA's churning field names (the load-bearing constraint).** The crate
  produces a **stable, substrate-shaped `AgentFacts` struct** (our names: identity, capabilities,
  locality, telemetry, certification, ttl) and a **thin serializer** maps it to the *current* NANDA
  JSON-LD field names. When the spec renames (e.g. AgentFacts → "Agent Metadata Layer"), only the
  serializer changes — the substrate-derived core never does. Never `#[serde(rename)]` the stable
  struct to a moving spec name.
- **F.3 — Self-certified only (Principle 1).** Ed25519-signed by the **node identity**; no
  issuer/TRS authority (that's a P-axis chokepoint / coordinator). The fetcher verifies the
  signature against the node's public key; trust is the caller's to decide.
- **F.4 — Dependency surfaced (the M16-A prerequisite).** Node-identity signing is currently
  internal (`NodeTls::signing_key`, `tls::sign_bytes`). A public-API companion crate can't
  self-sign under the node identity. **Add a public, `tls`-gated signing surface on `GossipAgent`**
  (`sign_with_identity(msg) -> Option<[u8;64]>` + `identity_public_key() -> Option<[u8;32]>`) — a
  small, clean surfacing of existing capability, analogous to closing WS-E's bulk-fetch-client gap.
  This is the kickoff PR.
- **F.5 — Public + verifiable, never token-gated** (ROADMAP precursor criterion). The AgentFacts
  endpoint stays outside the `/gateway` scope wall — publicly fetchable, cryptographically verified,
  TTL-scoped (`facts_url`). The field mapping is "mapping, not new mechanism" (ROADMAP §16 table).
- **F.6 — `evaluations`/`telemetry` are *self-attested-with-audit*.** Where the doc reports
  self-asserted claims, cite the WS2 audit trail's stable `content_hash` as provenance — turning
  *self-advertised* into *self-attested* (the precursor dependency ROADMAP §gap-7 called out).
- **F.7 — M16-B reuses the substrate as the CRDT.** Per-field `SignedData` KV writes + LWW + HLC +
  anti-entropy *are* the "CRDT update protocol" — no new transport; push intra-domain, pull at edge.

## Philosophy compliance — guardrails (validated 2026-06-15)

The plan was audited against the six Core Principles (no coordinator; consistency-as-service;
layer discipline; detection-not-prevention; emergent/threshold; composability) and the
Holland/Hayek grounding. WS-A, WS-E, WS-F, WS-G hold cleanly. Three items carried
prevention- or authority-shaped wording (mostly inherited from pre-v1.x-hardening ROADMAP
text) and have been **re-aligned in ROADMAP**; their compliant shape is recorded here as a
binding guardrail so it is not re-litigated at build time. Plus one meta-guardrail and one
watch-item.

1. **M6 (gossip capability authz) — enforce at resolve/serve, detect at write.** *No Layer I
   write guard, no forwarding-hop block* — that would teach the gossip layer a higher-layer law
   (the inversion the commit-conflict and `sys/` tripwires exist to avoid). The resolver gate on
   signed-role ACLs is the enforcement point (consumers route around unauthorized providers);
   write-time is detection only (tripwire + counter). The advertisement still propagates per LWW.

2. **M7 (rate-limiting) — shared observation, local decision.** Gossiped `sys/rate/` data is
   *evidence*, not a verdict. Each node independently tightens its own inbound budget on local
   evidence (Principle 5); disconnection is a node-local self-defense choice. **No cluster-wide
   consensus eviction verdict** — that is a coordinator enforcing a behavioural judgment.

3. **M9 (`ClusterTuner`) — advisor, not config coordinator.** A node applies a `sys/config/`
   recommendation *only if its local policy accepts it*; application is never mandatory. The tuner
   advises, the node decides (Principle 1) — otherwise it acquires standing agency over cluster
   config.

**Meta-guardrail — library, not platform.** v2's self-tuning / supervision / provisioning /
federation is exactly the feature set that pulled BRAIN-IoT into platform-hood. The defense is the
**Layer column** above: every platform-like behaviour ships as a **companion crate or app-layer
agent on the public API** (the tuple-space proof), never as substrate. The substrate stays a
library; the platform is an emergent composition the operator assembles. An item drifting from
**C/A** into **S** is a design smell to challenge.

**Watch-item — OR-Map CRDT ([Quilt-DD#3]).** Must be an *application-level interpretation layered
on LWW entries*, not a second substrate merge rule — otherwise it fractures "single KV substrate,
LWW is the law." The "design note, not a rewrite" framing is load-bearing.

**Net:** with these guardrails recorded, there are **no open philosophy conflicts** in the plan —
the three tensions convert from unreconciled wording into binding design contracts. They are
*addressed* at the design/contract level; they are *honoured* when each milestone is implemented
to the shape stated here. (This section, plus the per-milestone *compliant shape* in ROADMAP, is
the standing check.)

## Definition of done for v2.0

v2.0 is **delivered** when, for each workstream, *either* its **Done when** clause is met,
*or* the workstream is explicitly deferred-with-reason (its triggers never fired) and that
deferral is recorded here. The completeness matrix above must have **no orphaned theme** at
release.

This is deliberately an *acceptance gate*, not a schedule. Because v2 is demand-driven, a
legitimate v2.0 release may ship some workstreams and consciously defer others — what it may
**not** do is leave a discussed theme unaccounted for (neither built nor explicitly deferred).
