Guide — what to observe
Layer I The committed value is written to consensus/committed/config/epoch and gossip-replicated; all nodes converge to the same key regardless of who committed.
Layer III Competing group_propose calls — the first to gather ≥5/9 ACKs commits; the rest detect a higher ballot already accepted and receive Superseded.
  1. Click Start race — all 9 nodes fire simultaneously; watch one turn green (committed) while the others turn purple (superseded) within ~2 s.
  2. The committed banner updates as soon as the winner is known; all nodes then read the same value from Layer I KV via anti-entropy.
  3. Click Solo propose node 0 — with no competition, node 0 commits immediately on ballot 1 with no superseded peers.
  4. Race outcome order varies — ballot jitter (±80 ms) breaks ties and prevents livelock when two nodes reach quorum simultaneously.

How it works: All 9 nodes call group_propose simultaneously with different values. The first ballot to reach quorum (5/9) commits — the winner gets Committed. The rest receive Superseded when they detect a higher ballot already committed.
Competing proposals
First quorum wins
All 9 nodes call group_propose simultaneously. Each sends a PROPOSE with ballot 1 and its own value. Whichever node's ballot first accumulates ⌊9/2⌋+1 = 5 ACKs commits. The committed value is written to Layer 1 KV and epidemic-synced cluster-wide within milliseconds.
// All nodes propose simultaneously let result = agent.group_propose( "cfg_cluster", "config/epoch", Bytes::from("epoch-5-node-3"), cfg, ).await;
Committed
One value, all converge
The winner receives ConsensusResult::Committed. Even the losers (Superseded) immediately read the correct committed value via consensus_get — or watch it with consensus_rx for a push notification. Late-joining nodes get the committed value via anti-entropy KV sync.
// Loser reads the winner's value Superseded { slot, ballot } => { let v = agent .consensus_get(&slot) .map(|b| String::from_utf8_lossy(&b) .to_string()); // v == winner's committed value }
Ballot retry jitter
Breaking livelock
If two proposers collide on ballot 1, both retry with ballot 2. Without jitter they'd collide again infinitely. ballot_retry_jitter_ms = 120 inserts a random sleep before each retry so one node wins the next ballot unopposed. Watch the ballots counter: usually 1, occasionally 2+ when the race is tight.
ConsensusConfig { phase1_timeout: Duration::from_millis(1200), max_ballots: 3, ballot_retry_jitter_ms: 120, ..Default::default() }
