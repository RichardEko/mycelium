The New Reality02 / 25
AI agents are not microservices
The infrastructure playbook for microservices breaks down when your units of compute are ephemeral, capability-driven, and self-organizing.
Microservices World
Static topology
Services have fixed names and addresses. Service discovery finds them. They don't change shape at runtime.
Register → serve → deregister
Explicit lifecycle. The registry must be told when something starts and when it stops. Crashes leave stale entries.
Coordinator-centric
One system knows the topology. Everyone asks it. It becomes the bottleneck and the failure mode.
AI Agent World
Dynamic capability sets
An agent's role emerges from what it can do — and that changes as models load, tasks complete, and resources shift. Discovery must keep up.
Ephemeral by nature
Agents spawn, exhaust their purpose, and die. The fleet must heal around them automatically. No operator involvement.
Peer-to-peer intelligence
Agents need to find each other, negotiate capability, and coordinate — without a broker in the critical path that becomes a single point of failure.
The infrastructure hasn't kept up. The tools we have were built for stateless HTTP services, not for fleets of reasoning agents with dynamic capability surfaces.

The Claim·· / ··
One fabric. Every interaction pattern.
Most "agentic" frameworks give you one way for agents to interact — a remote call wearing a prompt, brokered by an orchestrator. But coordination is older and richer than RPC. Mycelium offers the whole taxonomy, and every shape is just reads, writes, and signals on the same coordinator-free substrate.
Request / Response
rpc_call · scatter_gather — an addressed call you await, or a fan-out to every provider of a capability with replies gathered. The one shape most agentic stacks stop at.
Publish / Subscribe
emit · subscribe — Signals (Layer II): ephemeral, scoped, admission-bounded broadcast. Fire an event; only nodes inside the boundary that recognise it react.
Actor Mailbox
deliver_event · open_mailbox — a durable, HLC-ordered, addressed inbox. A conversation that survives a restart and never reorders under clock skew.
Shared State / Stigmergy
set · subscribe_prefix — the gossiped LWW KV store is a shared environment. Agents coordinate by modifying it — load pheromones, soft-state intent — with no message at all.
Pull · Tuple Space
put · take — positional work distribution. Producers drop items into lanes; workers take when ready. No component models anyone's load.
Blackboard
post · claim — content-routed competitive claim. Agents post facts; others claim by an attribute predicate. Topology emerges from what the fact says.
…plus ballot consensus (propose) when you genuinely need agreement. Seven interaction shapes, one coordinator-free substrate — chosen per interaction, not dictated by the framework. The RPC monoculture of today's agentic stacks is a constraint Mycelium simply doesn't have: pick the pattern the problem wants, and it composes on the same gossip fabric as all the others.

Theoretical Foundations05 / 25
Not retro-fitted to theory.
Built from it.
Mycelium began with John Holland's Complex Adaptive Systems framework as its design blueprint — then asked what that looks like running in Rust at production scale.
Holland — Signals & Boundaries (MIT Press, 2012)
Signals
Information that travels between agents and triggers responses. Selective — only agents that recognise the signal act on it.
Boundaries
Constraints that define which agents interact. Boundaries create structure from the inside — no external authority assigns them.
Tags & Aggregation
Agents carry tags (declared properties). Matching tags cause spontaneous aggregation into higher-order structures — emergent groups with no coordinator.
Emergence over prescription
Macro-level behaviour arises from micro-level rules. The designer specifies interaction primitives, not outcomes.
Mycelium — The Implementation
Layer II — Signal Mesh
Scoped ephemeral events with admission boundaries. Only nodes that advertise the matching capability receive group-scoped signals. Holland's signals, made concrete.
Capability Groups & Opacity
Group boundaries emerge from capability declarations. Opacity composition enforces admission without a gatekeeper. The boundary is distributed, not policed.
Capability Advertisement (Tags)
advertise_capability("compute","gpu") is a tag. resolve() matches tags. Emergent groups self-assemble from nodes sharing compatible tag sets.
No coordinator, by design
The cluster topology is never prescribed. It emerges from gossip, TTL expiry, and capability matching — exactly as Holland's CAS model predicts it should.
Also standing on: Grassé (1959) Stigmergy — agents coordinate through environment modification, not direct communication. The gossip KV layer is the environment. · Demers et al. (1987) Epidemic Algorithms — the gossip substrate. · Kulkarni et al. (2014) Hybrid Logical Clocks — causal ordering without NTP.
And producing: the theory runs both ways — The Coordinator Trap (2026, submitted) argues coordinator-freeness is epistemically necessary, not merely convenient (a Hayekian local-knowledge argument), and a second paper formalises substrate convergence. The design isn't decorated with citations after the fact; it is an executable instance of a published argument.

Self-Organization07 / 25
The fleet knows
what it can do
Capabilities advertised, not registered
Each node writes its capabilities as TTL'd KV entries that gossip through the cluster. Groups form automatically from nodes sharing compatible capabilities. No administrator defines the groups.
Dead nodes vanish automatically
When a node stops refreshing its TTL entries, they expire. The cluster heals in under one health interval. No deregister call. No stale entries. No operator alert.
Late joiners catch up fully
Anti-entropy sync on reconnect means a restarted node recovers the entire cluster state from its peers within seconds. No replay log. No snapshot restore.
The cluster knows what it can do — and it always has an up-to-date view, on every node, without asking a coordinator.
Agentic Flow Networks08 / 25
The cluster is the queue
Conventional pipelines pin workers to static stages. AFN workers shift to wherever demand exists.
Opacity Backpressure — automatic
Busy workers mark themselves opaque. The capability resolver skips them. Overloaded stages draw in idle workers from elsewhere — zero config.
KV ring IS the pipeline buffer
Stage items are KV entries. Any worker that matches the capability can pick them up. No queue server, no topic config, no dead-letter queue.
Push → Pull — the last decision-maker, removed
The KV-ring buffer still lets a coordinator push work to a predicted worker. The mycelium-tuple-space companion flips this: workers take when ready, so no component models anyone's load. Linda's pull pattern, on a coordinator-free substrate. (Appendix A10)
Beyond the demo
The demo uses identical workers that each carry all stage logic — vivid, but it undersells the model. In production: specialist workers advertising only one stage, heterogeneous teams shipping independent capability sets, zero-downtime rollout of score_v2 alongside score_v1. The mesh assembles the topology from whatever is running. No operator involvement.
Demo assumption ≠ Mycelium's requirement. Fluid allocation works across a fully heterogeneous fleet — different images, different teams, different capabilities. Topology emerges bottom-up.
v2.0 · Metabolism·· / ··
The substrate runs itself.
"Zero ops" isn't "no processes to run" — it's a cluster that tunes, scales, and heals itself with no coordinator and no operator in the loop. Management is never a command; it is an intent every node reconciles locally.
Self-tuning from size
A node derives its own channel depths, intervals, and fan-out from the observed cluster size — closed-form, no tuning file. As the cluster grows or shrinks every node re-derives, and config hot-reloads live with no restart.
Elastic membership + self-heal
A MembershipGovernor reconciles a group toward a min/max band and drains nodes cooperatively. Kill the governor — the cluster still self-heals; the intent lives in evaporating soft-state, owned by no one.
Defends itself
Distributed rate-limiting: nodes share observed per-sender rates and each independently throttles an aggregate abuser to a fair share. A sustained flooder is clamped by every node it touches — with no global eviction verdict.
// Management = intent + local reconcile. No command. No coordinator. // Posted by anyone — a human or an agent — they are substrate-identical. POST /gateway/govern/membership { "group":"workers", "min":3, "max":12 } POST /gateway/govern/timing { "health_secs":2, "backoff_secs":1 } // → written as evaporating soft-state; every node reconciles toward it // → newest-wins, node-sovereign; kill the poster and the last intent holds
The litmus test. "If management vanishes, does the cluster self-heal?" Yes — because there was never a manager, only nodes reconciling a shared desired-state. Operators are optional; the deck's "zero ops burden" is this, not merely "no broker process."
v2.0 · Coordination·· / ··
Coordination, two ways.
Two companion crates — both built entirely on the public API — revive the Linda / JavaSpaces lineage: its two access modes, coordinator-free, minus the transaction monitor that made the original a single point of failure.
⬍
Tuple space — pull by position
mycelium-tuple-space. Producers put into named FIFO lanes; workers take when ready — Linda's in, a competitive exactly-once destructive claim. An item's lane is its pipeline stage; per-lane depth is the fluid workers' pressure signal. Keyed take_by_key adds O(1) fan-in joins.
▦
Blackboard — claim by content
mycelium-blackboard. Agents post facts; others read them non-destructively (Linda's rd) and claim a finite one competitively (in) — matched by an attribute predicate, not a stage. Which agent acts is decided by what the fact says, not where it sits.
The classical objection, dissolved. Gelernter's tuple space (1985) and JavaSpaces were always the right pull primitives; the standing complaint was that the shared space is itself a coordinator needing consistency machinery. Build the space on a coordinator-free substrate and that objection has no variable to inhabit — the ring is the failure detector, a standby self-promotes, acknowledged work never resurrects. Pull (position) and claim (content) are the two shapes — Mycelium ships both.
v2.0 · Code Mobility·· / ··
Capabilities that travel.
A node that can't satisfy a requirement doesn't page an operator — it provisions the capability itself. Content-addressed WASM components move over the mesh, verified, and a restart is indistinguishable from a first provision.
The autonomic loop
requirement → resolve → pull → verify → instantiate → advertise → serve → self-heal → shed. No coordinator assigns work; each node closes the loop locally against the gossiped catalogue. The classic requirement→deploy loop, done as level-triggered soft-state.
Content-addressed + signed
An installable/ entry is a hash + Ed25519 provenance; bytes pull over the mesh — no registry server, the catalogue is the gossip store — and verify identically on every node. WASM runs fuel-limited (mycelium-wasm-host / wasmtime): the OSGi-bundle analogue Rust never had.
// Publish once — gossips to every node, no registry to run let entry = InstallableEntry::new(ns, name, artifact_id) .provides(["tool/translate"]).signed_by(&key); publish_installable(&agent.kv(), &entry); // A node missing a requirement self-provisions: // resolve installable/ → pull bytes over the mesh → verify // hash + sig → instantiate WASM → advertise the capability // Restart ≡ provisioning. Failure ≡ a fresh resolve.
Federation, shipped. Across domains a node publishes self-certified AgentFacts — a NANDA-shaped, Ed25519-signed quilt-patch — plus a CRDT-assembled domain view: sovereign discovery with no issuer authority. The internet-scale tier is no longer a future box on the stack.

Three-layer Architecture
Layer I — KV Store
Last-write-wins gossip over TCP. HLC causal ordering. TTL per key. Anti-entropy on reconnect. Bloom-filtered dedup. WAL persistence.
Layer II — Signal Mesh
Ephemeral scoped events. Broadcast / group / individual scopes. Opacity composition for admission control. Pheromone attenuation at group boundaries.
Layer III — Consensus
Epidemic ballot-voting (CASPaxos-adjacent). No Raft leader. consistent_set, distributed_lock, elect_leader. Cross-group proposals via GroupQuorum.
Single KV substrate. All three layers write to dedicated key prefixes via the same gossip mechanism. cap/ · sig/ · consensus/ · sys/ · mailbox/ · prompts/ — one substrate, many namespaces.
Structural Inversion: AP substrate always available. CP overlay opt-in. Pay for ordering guarantees only where your application demands them.

Consensus — Ballot-Serialized Agreement
Epidemic ballot voting — no leader
CASPaxos-adjacent. Proposals gossip through the cluster. No Raft leader. Concurrent writes to the same key are totally ordered by ballot number.
Cross-group federation
cross_group_propose + GroupQuorum — multiple groups each act as an independent voting bloc. Models multi-AZ durability gates and hierarchical approval.
Why not Raft: Raft requires a stable leader. Agent fleets have constant churn. Epidemic consensus tolerates node turnover without stalling progress.
// Ballot-serialized KV write agent.consensus().consistent_set("key", val).await?; let v = agent.consensus().consistent_get("key"); // Distributed lock service — blocking acquire + monotonic fencing token let lock = agent.consensus().locks() .lock("job-42", ttl, wait).await?; stamp_resource(lock.token); // resource rejects a lower token → stale writers fenced // Leader election let leader = agent.consensus().elect_leader("shard-0").await?; // Cross-group proposal — both blocs must reach quorum agent.consensus().cross_group_propose("config", val, vec![GroupQuorum { group: "llm", quorum: 0.5, veto: false }, GroupQuorum { group: "compliance", quorum: 0.5, veto: true }]).await?;

Security — Transport · Identity · Integrity
mTLS transport --features tls
Mutual TLS on every gossip TCP connection. External attackers cannot inject or intercept frames. Zero overhead when the feature flag is off — all TLS code compiles away.
Ed25519 node identity
Each node generates an Ed25519 keypair at startup. Same keypair serves as mTLS identity, gossips into sys/identity/{node}, and signs consensus payloads.
Signed KV writes
WireMessage::SignedData — Ed25519 signature over key, value, timestamp, tombstone flag, and sender. A compromised node cannot inject false KV entries.
Compliance layer --features compliance

Appendix A1025 / 25
TupleSpace — Pull, Without the Coordinator
A companion crate (mycelium-tuple-space) built entirely on the public API — the constructive proof that the most coordinator-shaped workload, work distribution, needs no decision-maker.
Pull, not push
Producers put items into named stages; workers block on take and receive work exactly when ready. Readiness is announced by the claim itself — there is no model of anyone's load to go stale, so the misrouting that staleness causes in push systems has no variable to inhabit.
The ring IS the failure detector
The serving node is discovered by capability advertisement; its failure is detected by the same TTL evaporation that fades any pheromone; a mirroring standby promotes itself through the same emergent election as every other role. Kill the primary mid-flight and survivors are served — acknowledged items never resurrect.
Durable by construction
Single-lock hot path; a write-ahead log whose Complete record is one indivisible entry, so a stage transition can never half-replay. Compaction bumps a WAL epoch so a secondary's byte-offset replay cursor can never silently dangle.
// Producer — no knowledge of any worker let id = space.put("stage.fetch", payload).await?; // Worker — blocks until work is ready, then claims it let (id, item) = space.take("stage.fetch").await?; // ...do the work... space.complete(id, "stage.parse", next).await?; // atomic ack + advance
Why it matters. Gelernter's tuple space (Linda, 1985) was always the right pull primitive; the classical objection — the shared space is itself a coordinator needing consistency machinery — dissolves when the space is built on a coordinator-free substrate. 1,000 items · 10 producers · 10 workers · exactly-once · zero shared load view. Side note: what is kept is pull + decoupling, not Linda's associative matching — the space is lane-addressed (named per-stage FIFO lanes, opaque payloads, O(1) claims), and the per-lane depths double as the fluid workers' pressure signal.
Composability proof. The crate's only dependency on the core is the public API — pull composes from the existing primitives. Nothing was added to the substrate to make it possible. Worked example: examples/fluid_pipeline (the AFN pull pipeline — coordinator/worker over the tuple space, both pull and push modes in ci_smoke.sh).

Blackboard — Claim by Content
A companion crate (mycelium-blackboard), public-API-only — the content-routed sibling of the tuple space. Where the tuple space routes by lane position, the blackboard routes by a predicate over fact attributes.
The one new primitive: claim(predicate)
Competitive, destructive claim-by-predicate (Linda's in): a finite fact matching the predicate is claimed by exactly one agent — single-owner, non-blocking (the loser gets None, no parked waiters). Non-destructive read (rd) is shared/concurrent. ack is the idempotent terminal; an in-flight deadline re-queues (at-least-once).
Predicate, not unification
The match language is the capability attribute-filter grammar — equality + presence (Predicate::new().eq(k,v).present(k)) — deliberately not template/unification matching. Topology emerges from what a fact says.
Mirror-complete replication
Post/Ack-only — the deliberate divergence from the tuple space. A claim doesn't change a mirror's liveness, so a claimed-but-unacked fact stays claimable on a standby = the at-least-once re-queue a promotion wants. No heartbeat, no WAL-replay cursor.
// Post a fact — attributes + payload, no addressee board.post(BTreeMap::from([ ("kind".into(), "repair".into()), ("zone".into(), "north".into())]), payload).await?; // Read shared (rd) — non-destructive, concurrent let open = board.read(&Predicate::new().eq("kind","repair")).await?; // Claim (in) — competitive, exactly one winner if let Some(f) = board.claim(&Predicate::new() .eq("zone","north").present("kind")).await? { // ...do the work... board.ack(f.id).await?; // idempotent terminal }
The exactly-once contract, reused. The blackboard is the second real user of the WAL claim/ack/requeue discipline (the tuple space is the first). The shared overlay was examined and declined-with-evidence — they diverge on a load-bearing axis (wall-clock-ms/cross-node vs monotonic/in-process). The shared artifact is the contract, not the code. Worked example: mycelium-blackboard/examples/microgrid.

Mailbox & Events — the Actor Inbox
Signals are ephemeral broadcast; sometimes an agent needs a durable, ordered, addressed inbox. MailboxHandle is that primitive — an actor mailbox on the substrate, with no broker behind it.
Durable + HLC-ordered
Messages land in a per-recipient mailbox backed by the KV store, delivered in Hybrid-Logical-Clock order — so causal happens-before survives wall-clock skew. Survives restart; a late reader catches up via anti-entropy like any other state.
Two delivery shapes, one mesh
Signals (Layer II) — ephemeral, scoped broadcast for events: fan-out, fire-and-forget, admission-bounded. Mailbox — durable, point-to-point, ordered: the actor inbox. Pick per interaction; both ride the same coordinator-free substrate.
The actor ↔ LLM pattern
The canonical use (coop demo 01): an actor and an LLM worker exchange turns through the mailbox, so a slow or restarted model never drops or reorders a conversation. MeshEvent is the typed envelope.
// Open a durable, ordered inbox → (handle, receiver) let (_mbox, mut rx) = agent.service().open_mailbox("triage.ask", 64); // Send — addressed by NodeId + kind only, no broker agent.service().deliver_event(&target_id, "triage.ask", payload); // Receive — drained in HLC-causal order, durable across restarts while let Some(ev) = rx.recv().await { // ev: MeshEvent { kind, sender, payload, hlc_ts } }
Why not just signals? Signals are the right tool for ephemeral events — but an agent conversation needs delivery and order. The mailbox gives both without standing up a queue: the durability is the KV store, the ordering is the HLC, the addressing is the capability ring. The actor model, with nothing to operate. Worked example: examples/coop/src/bin/mailbox_llm (an actor and an LLM worker exchange turns through the mailbox — coop demo 01).

Wiki — Durable Curated Knowledge
A companion crate (mycelium-wiki), public-API-only — the durable sibling of the transient work-finding primitives. Tuple space and blackboard route work to be consumed; the wiki is the group's maintained-meaning canon: where a group keeps what it learns.
Control plane / data plane
The corpus is not in gossiped KV — it lives in a node-independent, pluggable store (shared FS / S3 / doc store). A single elected curator serialises writes; every agent reads the store directly, in parallel — no curator on the read path. Curator election + ring-failover ride the capability ring; if the curator vanishes the wiki stays readable and a new one self-elects.
Single writer, no CRDT
The curator groups proposals by section and reconciles each batch as one writer — the append-merge is lossless + idempotent, or an LLM 3-way merge behind the llm feature (prose curated, structure code-owned). A change-driven lint (dead cross-links, self-consistency) runs only after a write — an idle wiki does no work.
Composes, doesn't replace
The authoritative/maintained layer alongside Postgres (typed metrics) + RAG (background), joined by a shared id namespace — not a similarity index. Reached as MCP tools (wiki.read/query/propose), an HTTP gateway (/gateway/wiki/* + py/ts SDKs), or a membership-gated access broker.
// Propose an edit — the curator reconciles + applies it wiki.propose("decisions/elm-street", sid, "Resolution 2026-14", "protected bike lane approved", attrs); // Read directly from the store — no curator on the read path let page = wiki.read("decisions/elm-street")?; // One-time membership-gated grant, then read direct forever let grant = wiki.request_store_access().await?; // StoreGrant { location }
The wiki pattern, recursively. Files + an LLM curator + direct reads — exactly how Mycelium's own docs/wiki/ works. The concurrent-prose-merge problem dissolves into single-writer-curator + a dumb store. Worked example: mycelium-wiki/examples/wiki_chat — import documents, then chat grounded in the wiki (one template for the org-twin and council use cases).
