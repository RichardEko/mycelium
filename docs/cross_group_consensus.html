The API Gap

Deferred

Mycelium's consensus API currently offers two scopes. There is a gap between them that neither covers:

Exists
group_propose
Votes from all members of one named capability group. Quorum is a fraction of that group.
Missing — Phase 8
cross_group_propose
Votes tallied independently per group. Commits only when every group reaches its required quorum. Single ballot round — atomic, not a chain of separate calls.
Exists
system_propose
Votes from all nodes in the cluster. One large group can dominate a vote.
⚠
Why not chain N group_propose calls? Chaining is not atomic. Group A could commit while Group B times out — the slot value is then inconsistent across groups. A single ballot round where all groups' votes are collected and tallied independently by the proposer is both simpler and correct.
🎯

Use Cases & Applicability

Design
🌍
Multi-AZ Durability Gates

Require quorum from both an az-east group and an az-west group before committing a persistent state change. A partitioned AZ causes the proposal to wait rather than committing with only one AZ's acknowledgement.

Infrastructure
🤖
Multi-Stakeholder AI Pipelines

An LLM inference fleet (group: llm-workers) and a retrieval fleet (group: retrievers) must both agree before a new retrieval schema is deployed — a unilateral commit by one group breaks the other.

AI Agents
✅
Human-in-the-Loop Ratification

A proposers group submits a value; a reviewers group must ratify it. The reviewers group can be given a veto: true flag — unanimous rejection blocks the proposal regardless of the proposers' votes.

Governance
⚖️
Compliance & Regulatory Gates

A trading group of execution agents cannot commit a new strategy without quorum from a compliance group of audit agents. Compliance holds an effective veto — the system is auditable by construction.

Finance / Legal
🏢
Multi-Department Agent Fleets

In a large organisation with separate agent clusters per department, a shared resource allocation requires buy-in from each affected department's group — no single large department can dominate the vote by sheer member count.

Enterprise
🛡️
Byzantine Fault Isolation

If one group is partially compromised, requiring independent quorum from a second group ensures the compromised group alone cannot force a commit. Each group's quorum requirement acts as an independent safety gate.

Security
🔧
Operator / Worker Pipeline Reconfig

In Mycelium's own AFN pipeline pattern, coordinator nodes and worker nodes form distinct groups. A pipeline reconfiguration (stage count, timeout policy) requires sign-off from both — coordinators own the topology, workers must confirm they can satisfy the new requirements.

AFN
🗳️
Hierarchical Voting Authority

Different groups carry different quorum thresholds — a core group may require 2/3 supermajority while an advisory group requires simple majority. The proposal cannot pass without satisfying both independently.

Proposed API

API 
/// Per-group quorum requirement for a cross-group proposal.
pub struct GroupQuorum {
    pub group:  String,
    /// Fraction of non-opaque group members required to accept.
    /// Default 0.5 = simple majority.
    pub quorum: f32,
    /// If true: unanimous rejection by this group vetoes the proposal
    /// regardless of other groups' votes. Use for compliance / ratification.
    pub veto:   bool,
}

impl GossipAgent {
    /// Propose a value that commits only when ALL specified groups
    /// reach their individual quorum requirements.
    pub async fn cross_group_propose(
        &self,
        slot:   &str,
        value:  Bytes,
        groups: Vec<GroupQuorum>,
        config: ConsensusConfig,
    ) -> Result<ConsensusResult, ConsensusError>
}rust

Example — compliance veto pattern

let result = agent.cross_group_propose(
    "strategy/v42",
    new_strategy_bytes,
    vec![
        GroupQuorum { group: "trading".into(),    quorum: 0.51, veto: false },
        GroupQuorum { group: "compliance".into(), quorum: 0.67, veto: true  },
    ],
    ConsensusConfig::default(),
).await?;rust

HTTP Gateway

# POST /gateway/consensus/cross_group_propose
{

Protocol Design

Design

Ballot broadcast

The proposer broadcasts to the union of all specified groups' members. This requires a one-line addition to signal.rs:

// signal.rs — new variant
SignalScope::Groups(Vec<Arc<str>>),

// admits() — new branch
SignalScope::Groups(names) => names.iter().any(|n| self.groups.contains(n)),rust

Each ballot message carries the full groups specification so every recipient knows which voting blocs are involved. No other wire format changes needed — the existing WireMessage::ConsensusVote is unchanged.

Vote tallying (proposer side)

// For each incoming vote from node N:
for group in groups.iter().filter(|g| node_is_member_of(N, g)) {
    tally[group].accepts += 1;  // or .rejects
}

// Commit condition — ALL groups must pass:
let commit = groups.iter().all(|g| {
    let fraction = tally[g].accepts as f32 / group_size(g) as f32;
    fraction >= g.quorum
});

// Veto condition — unanimous rejection by a veto group:
let vetoed = groups.iter().any(|g| {
    g.veto && tally[g].rejects == group_size(g)
});rust
✓
What does not change: the ballot wire format, ConsensusResult type, group_propose, system_propose, or the ConsensusEngine receive path. The proposer-side grouping is post-processing on vote data already available.

Timeout and failure handling

Same ConsensusConfig.timeout_ms governs the entire ballot. Opaque nodes are excluded from each group's denominator exactly as in group_propose today — the min_members guard applies per-group.

When to Implement

Deferred

Deliberately shelved — the existing group_propose and system_propose cover all known current needs. The implementation is modest (~150 lines); the deferral is about priority, not complexity.

💡
Triggers to revisit:
  • A multi-AZ deployment where per-AZ quorum is a safety requirement
  • A human-in-the-loop AI workflow where a compliance group needs veto rights
  • A multi-department fleet where no single group should dominate a cluster-wide vote
