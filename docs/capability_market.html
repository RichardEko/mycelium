Capabilities — demand pressure
Guide — what to observe
  1. Each requirer tile shows Satisfied when at least one provider matches its (ns, name). Disable the only matching provider and the tile flips to Unsatisfied within ~50 ms — the C2 reconcile-debounce window.
  2. Enable a backup provider — every requirement of that kind snaps back to Satisfied as the cap/ gossip lands at the requirer.
  3. The middle column reads GossipAgent::demand(filter) on each tick: green = pressure ≤ 1, amber ≤ 2, red > 2 (demand exceeds supply).

Event log
advertise_capability
Provider lifecycle
A node writes cap/{self}/{ns}/{name} for every capability it exposes and the handle re-asserts the key on an interval. Drop the handle and the entry is tombstoned — every interested watcher sees the change via gossip.
let h = agent.advertise_capability( Capability::new("compute", "gpu"), Duration::from_secs(30), ); // drop(h) → tombstone cap/{self}/compute/gpu
watch_requirement
Push-based satisfaction
The requirer subscribes once and receives a RequirementStatus every time a matching provider appears or disappears. C1 narrows the underlying prefix watcher to this filter's (ns, name); C2 coalesces bursts into a single update.
let mut rx = agent.watch_requirement( CapFilter::new("ai", "agent"), ); loop { match &*rx.borrow_and_update() { Satisfied { providers } => { .. } Unsatisfied { filter } => { .. } } rx.changed().await?; }
demand
Pressure as a first-class signal
demand_pressure = demanders / max(providers, 1). The library never auto-scales — orchestrators read this surface and decide whether to spawn another provider, rebalance, or alert. Each capability card here reflects that read.
let d = agent.demand(&filter); // d.demanding_nodes: Vec<NodeId> // d.providers: Vec<NodeId> // d.demand_pressure: f32 if d.demand_pressure > 1.5 { // scale up … }
