Guide
  1. The consumer fires resolve_with_locality(&filter, PreferShared(0)) every 500 ms. The returned list is ordered by shared-prefix depth descending; the first entry wins.
  2. Click the provider in the consumer's same rack to disable it — the chosen pip jumps outward to the next-closest ring.
  3. Click a non-provider node to make it the consumer. The provider depths recompute against the new vantage point; ring positions rearrange.
  4. Disable every provider — the resolver returns an empty list and the Chosen stat reads — until you bring one back.
Event log
LocalityPath
Topology address per node
Set via config.locality_path at agent construction. The library publishes it as cap/{self}/locality/self on start(). Peer paths are cached in kv_state.peer_localities and pre-warmed from KV so cold resolves don't silently score everything at depth 0.
let mut cfg = GossipConfig::default(); cfg.locality_path = vec![ "az1".into(), "rack0".into(), "host2".into(), ];
resolve_with_locality
Depth-aware provider ordering
Returns every cap/ match annotated with its shared-prefix depth against this node's locality. The preference re-orders or filters: PreferShared(d) sorts by depth descending, Strict(d) drops anything below d.
let matches = agent.resolve_with_locality( &CapFilter::new("render", "job"), LocalityPreference::PreferShared(0), ); // Vec<(NodeId, Capability, depth)>, // closest first.
why this matters
Same-rack > same-az > remote
A node in az1/rack0/host2 reaches a same-rack provider with two shared segments (depth 2), a cross-rack same-az provider at depth 1, and a different-az provider at depth 0. The library doesn't impose policy — it surfaces depth so the caller can decide between latency, fault isolation, egress cost, or just do not leave the rack.
// Strict: must be in the same az or fail let only_same_az = agent.resolve_with_locality( &filter, LocalityPreference::Strict(1), );
