Turn a TOML file into a mesh node

ℹ️
Two paths to LLM skills on Mycelium:

SkillRunner (this doc) — no Rust required. Define your skill in a .skill.toml manifest, point it at any OpenAI-compatible LLM, and run the binary. Best for deploying standalone agents or extending an existing mesh without touching library code.

Prompt Skills (--features llm) — library API. Call register_prompt_skill / call_prompt_skill from Rust; templates live in the KV ring at prompts/{ns}/{name}; Python and TypeScript callers reach them through the HTTP gateway at /gateway/llm/call and /gateway/llm/stream. Best when you are already embedding Mycelium and want LLM capability without a separate binary.
⚡

Quick start

Guide

Build the binary, write a manifest, run it.

# build once
cargo build --bin skillrunner

# run (requires Ollama or any OpenAI-compatible server)
./target/debug/skillrunner --skill examples/skills/hello.skill.toml

# smoke test from a second terminal
cargo run --example invoke_skill
sh

The skillrunner node starts, advertises llm/hello on the mesh, and waits for rpc_call("skill.invoke", json_payload) invocations. The invoke_skill example discovers it and calls it end-to-end.

💡
Community example: examples/community/ contains a 3-skill orchestrator → researcher → writer pipeline with ./start.sh and ./invoke.sh "your topic".

LLM compatibility

Reference

SkillRunner speaks the OpenAI /v1/chat/completions protocol. Any server that implements this endpoint works:

Ollama
llama.cpp
LM Studio
vLLM
Jan
OpenAI
Anthropic*
Groq

* Anthropic requires an OpenAI-compatible adapter (e.g. anthropic-sdk-compat).

Tool calling (for skill composition) requires the server to support tool_choice: "auto" and the standard tool_calls response field. All servers in the list above support this except older llama.cpp builds — upgrade to llama-server ≥ b3000.

Reference

Invoking from Rust

Guide

Any GossipAgent on the mesh can discover and call a skill. The full example is in examples/invoke_skill.rs.

use std::time::Duration;
use bytes::Bytes;
use mycelium::{CapFilter, GossipAgent, GossipConfig, NodeId};

// 1. Start a caller node
let node_id = NodeId::new("127.0.0.1", 7955)?;
let mut cfg = GossipConfig::default();
cfg.bind_port = 7955;
cfg.bootstrap_peers = vec![NodeId::new("127.0.0.1", 7947)?];
let agent = std::sync::Arc::new(GossipAgent::new(node_id, cfg));
agent.start().await?;

// 2. Discover the skill
let filter = CapFilter::new("llm", "chat");
let (skill_node, _) = agent.resolve(&filter)
    .into_iter().next()
    .expect("no skill on mesh");

// 3. Invoke it
let payload = serde_json::to_vec(&serde_json::json!({
    "message": "What is a gossip protocol?"
}))?;
let result = agent.rpc_call(
    skill_node, "skill.invoke",
    Bytes::from(payload),
    Duration::from_secs(30),
).await?;

let reply: serde_json::Value = serde_json::from_slice(&result)?;
println!("{}", reply["reply"]);
rust

Invoking from Python

Guide

The mycelium-py bridge connects to any node's HTTP gateway (http_port in [node]). Set http_port = 9000 on the skillrunner node and connect Python to it.

import json
from mycelium import MyceliumAgent

# Connect to the HTTP gateway of any mesh node
agent = MyceliumAgent("127.0.0.1", 9000)

# Discover
providers = agent.resolve_capability("llm", "chat")
node_id   = providers[0].node_id

# Invoke
payload = json.dumps({"message": "Hello!"}).encode()
result  = agent.rpc_call(node_id, "skill.invoke", payload, timeout_secs=30)
print(json.loads(result)["reply"])
python
# pip install -e mycelium-py/
cd mycelium-py
pip install -e ".[dev]"
sh

Invoking from TypeScript

Guide

The mycelium-ts package exposes the same gateway API as the Python bridge. Set http_port = 9000 on the skillrunner node and connect Node.js to it.

import { MyceliumAgent } from "mycelium-ts";
import { Buffer } from "buffer";
typescript


// Connect to the HTTP gateway of any mesh node
const agent = new MyceliumAgent("127.0.0.1", 9000);

// Discover
const providers = await agent.resolveCapability("llm", "chat");
const nodeId = providers[0].nodeId;

// Invoke
const payload = Buffer.from(JSON.stringify({ message: "Hello!" }));

console.log(JSON.parse(result.toString()).reply);

Skill composition

Guide

A skill's tools array names other mesh capabilities the LLM may invoke. SkillRunner resolves them from the KV store at call time, hands their schemas to the model as OpenAI function tools, and dispatches any tool_calls the model returns back through the mesh as nested rpc_call("skill.invoke", …).

[skill]
prompt = "Research the topic then write a polished article."
tools  = ["llm/researcher", "llm/writer"]   # resolved at call time
toml

Call graph

Caller
└─ rpc_call("skill.invoke") ──▶ llm/orchestrator (port 7950)
├─ tool_call ──▶ llm/researcher (port 7952)
│ └─ LLM inference → result ──────────────────────┐
└─ tool_call ──▶ llm/writer (port 7953) ◀────────┘
└─ LLM inference → result ──▶ orchestrator ──▶ Caller
ℹ️
Load balancing is automatic. If two nodes both advertise llm/researcher, resolve() returns both. SkillRunner picks the first result; add CapFilter::with_ranking in library code to route by attribute.
⚠️
Depth limit: the tool-call loop runs up to 5 rounds per invocation. Deep chains work if the model converges quickly. If it issues more than 5 tool calls without a final response, the runner returns {"error": "exceeded maximum tool-call rounds"}.
⚖️

Concurrency and load management

Reference

max_concurrent limits simultaneous in-flight LLM calls per node. Requests beyond the limit receive {"error": "skill saturated"} immediately.

[capability.policy]
max_concurrent = 2   # GPU-bound: only 2 parallel inferences
toml

The mesh sees saturation automatically. When all slots are occupied, the node's load state is visible via sys/load/{node}/… in the gossip KV store. Upstream callers using is_node_opaque() or the demand subsystem route around it.

To scale out, run a second instance with the same [capability] block on a different bind_port. resolve() returns both; callers automatically distribute load.

Audit trail

Reference

Every invocation writes one record to the gossip KV store. Because the record is gossiped, every node in the cluster can read it.

Key format

audit/{unix_nanos}/{node_id}  →  JSON
key

Lexicographic key ordering gives time-sorted prefix scans across the whole cluster.

Record schema

FieldTypeDescription
skill_ns / skill_namestringThe invoked capability
callerstringNodeId of the requesting node
nonceu64RPC correlation nonce — use as trace ID to correlate with OTEL spans
successboolWhether the LLM call completed without error
duration_msu64Wall-clock time of the full invocation including tool calls
tool_callsarrayNames of sub-skills invoked during tool-call rounds
ts_unix_nanosu128Invocation timestamp (also the key prefix)

Reading audit records from Rust

let records = agent.scan_prefix("audit/");
for (key, val) in &records {
    let rec: serde_json::Value = serde_json::from_slice(val)?;
    println!("{}: {} ms  success={}",
        rec["skill_name"], rec["duration_ms"], rec["success"]);
}
rust

OTEL span export

Reference

Build with --features otel and add [skill.otel] to the manifest. Each invocation emits one OTEL span exported via OTLP gRPC.

# Build with OTEL support
cargo build --bin skillrunner --features otel
sh
[skill.otel]
endpoint     = "http://localhost:4317"   # OTLP gRPC collector
service_name = "my-skill"
toml

Span attributes

AttributeValue

💡
The span's trace ID is derived from the RPC nonce. Audit KV records and OTEL spans cross-correlate on the nonce field — one lookup finds both.

TLS and persistence

Reference

mTLS

Set [node.tls] to enable mutual TLS on the gossip TCP port. On first run SkillRunner auto-generates a cluster CA and a per-node certificate in auto_cert_dir. All nodes in the cluster must share the same CA.

[node.tls]
auto_cert_dir = "./mycelium-tls/"   # shared CA, unique cert per node

# Or supply externally-issued certificates:
# cert_pem    = "/etc/mycelium/node.crt"
# key_pem     = "/etc/mycelium/node.key"
# ca_cert_pem = "/etc/mycelium/ca.crt"
toml

KV persistence

Enables WAL + snapshot durability. The node survives unclean shutdown and replays its KV state on restart. Data lives under {base_path}/{node_id}/kv/.

[node.persistence]
base_path  = "/var/lib/mycelium/my-skill"
sync_flush = true   # fdatasync on every WAL write — safest
toml

Running a community

Operations

Each SkillRunner process is one node. Run multiple processes — each with its own bind_port and overlapping bootstrap_peers — and the mesh assembles itself through gossip.

# 3-skill community — see examples/community/
./skillrunner --skill examples/community/orchestrator.skill.toml &   # port 7950
./skillrunner --skill examples/community/researcher.skill.toml   &   # port 7952
./skillrunner --skill examples/community/writer.skill.toml       &   # port 7953

# Or use the provided script:
cd examples/community && ./start.sh
./invoke.sh "gossip protocols in distributed systems"
./stop.sh
sh
💡
Adding a second researcher node is automatic load-balancing. Both nodes advertise the same llm/researcher capability; the orchestrator's resolve() call returns whichever responds first.

The mesh converges via gossip within one health interval (default 10 s). For a faster initial convergence, set all nodes' bootstrap_peers to include one stable seed node (the orchestrator is a natural choice).

🔢

Port conventions

Operations

Each SkillRunner needs a unique bind_port. Suggested ranges:

7940–7949
Primary mesh seed nodes and non-skill services
7950–7999
SkillRunner skill nodes
9000–9099
HTTP gateways (http_port)

Environment variable overrides

All GossipConfig fields can be overridden without touching the manifest:

GOSSIP_BIND_PORT=7951 GOSSIP_BOOTSTRAP_PEERS="10.0.0.1:7946" \
    ./skillrunner --skill my.skill.toml
sh

Troubleshooting

Operations
{"error": "HTTP error: ..."}

The LLM server returned an error or was unreachable.

  • Check Ollama is running: curl http://localhost:11434/api/tags
  • Check the model is pulled: ollama pull llama3.2
  • Check endpoint in [skill.llm] — no trailing /
RUST_LOG=skillrunner=debug ./skillrunner --skill my.skill.toml
sh
timed out waiting for capability on mesh

The skill node isn't visible to the caller.

  • Check the skill process is running and the port is open: nc -zv 127.0.0.1 7950
  • Check bootstrap_peers in the caller's config points at the skill's bind_port
  • Wait one TTL interval (default 60 s) for gossip to converge, or lower ttl_secs
RUST_LOG=mycelium=debug ./skillrunner --skill my.skill.toml
sh
{"error": "skill saturated"}

All max_concurrent slots are occupied.

  • Increase max_concurrent in [capability.policy]
  • Run a second instance on a different port — the mesh load-balances automatically
  • Add retry logic in the caller with brief exponential back-off
{"error": "exceeded maximum tool-call rounds"}

The LLM issued more than 5 tool calls without producing a final response.

  • Add an explicit instruction to the prompt: "Make at most 2 tool calls then return your final answer."
  • Reduce the number of tools in the tools array
  • Use a more instruction-following model (e.g. qwen3:14b instead of llama3.2)
Port already in use

Each SkillRunner instance needs a unique bind_port. Check for leftover processes: lsof -i :7950. Use ./stop.sh or pkill -f skillrunner to clean up.
