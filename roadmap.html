Design Philosophy

🧬
Chemical Signalling

Hormones flood the bloodstream. Every cell receives every signal. Specificity comes from receptors — not from routing. This platform works the same way: signals flood the cluster epidemically; boundaries decide who acts. The metaphor isn't decorative — it's doing real architectural work.

⚡
Forwarding ≠ Acting

A node outside group::nlp forwards a group-scoped signal at full speed without acting on it. Propagation is guaranteed regardless of whether any intermediate node is in scope. No routing bottlenecks, no partial delivery.

🐜
Stigmergy

Workers write load state into the shared KV medium as pheromone trails. Routing emerges from nodes reading that medium — not from explicit coordination. Stale trails evaporate via embedded timestamps. New joiners bootstrap immediately via anti-entropy sync.

🔌
Threshold Activation

Adaptive systems act when they have sufficient local information — not when all agents are ready. last_signal, quorum, and watch implement this correctly. Barriers are an anti-pattern here: they block rather than degrade gracefully.

🎛
Serialisation Autonomy

Each agent picks its payload format independently — JSON for MCP compatibility, bincode for internal speed, protobuf for external contracts. The substrate carries opaque Bytes and routes by kind string. No cluster-wide migration when one agent upgrades its format.

🕸
Emergent Wiring

Nodes declare what they provide (cap/) and what they need (req/). The mesh resolves connections at the moment of use — no static "A connects to B" configuration, no service registry to keep in sync. Topology emerges from live capability advertisements and evaporates when nodes disappear. Autoscalers and orchestrators read demand_pressure as a continuous supply/demand signal rather than polling discrete health endpoints.

Deployment Model — Library, Not Platform

Understanding what Mycelium is determines who owns what and how a fleet is observed.

📦
It is a linked library

Mycelium is compiled into your binary. There is no Mycelium daemon, no control plane process, no central broker to operate. When the process starts, the node joins the mesh. When it stops, it leaves. Deployment is entirely the operator's concern — Kubernetes, AWS ECS, GCP Cloud Run, Puppet-managed VMs, bare metal; Mycelium has no opinion.

🔵
A cluster is emergent

There is no cluster object to create or join. A "cluster" is simply the set of process instances that embed Mycelium and can reach each other over TCP. Topology is the product of network reachability and seed-peer configuration — nothing else. Two groups of nodes that cannot reach each other form two independent clusters automatically.

🧱
Each cluster is isolated by construction

There is no cross-cluster Mycelium view. The management dashboard (port 8090) is itself just another node in the same cluster — it sees exactly what the gossip mesh exposes and nothing beyond it. If you run ten independent services, each embedding Mycelium, you have ten independent clusters with no shared state.

📡
Fleet-wide visibility is your stack's job

Datacenter or fleet-wide observability comes from wiring each cluster's metrics and log endpoints into the operator's existing tooling — Prometheus scraping each cluster's /metrics, Grafana aggregating across them, Datadog, Dynatrace, or whatever is already in place. Mycelium's Layer 5 work (planned) provides the scrape-compatible metrics endpoint each cluster exposes; aggregation and alerting remain in the operator's stack.

The closest analogues are Hazelcast embedded and etcd embedded: you ship the distributed system as part of your application binary, not as a separate service your application depends on. The operator's job is network reachability and process lifecycle — not operating a Mycelium cluster as an independent infrastructure tier.

The Structural Inversion: Consistency as a Service, Not a Foundation

Mycelium's defining architectural decision — and the reverse of nearly every production distributed system built in the last two decades.

⛓
How Raft systems work

Consistency is the foundation. Every operation flows through the consensus log. Consul, etcd, CockroachDB, TiKV. The benefit: strong guarantees everywhere. The cost: everything pays consensus latency, including the 95% of operations that don't need it.

📦
How Kafka works

The broker log is the foundation. Every event pays broker round-trip and partition coordination — including ephemeral signals that are immediately processed and discarded.

📮
How Akka works

The actor model is the foundation. Every message flows through a mailbox and supervision tree — including fire-and-forget notifications between co-located agents.

🌿
How Mycelium works

The epidemic gossip substrate is the foundation — always available, sub-millisecond, zero coordination overhead. Consistency, ordering, and reliable delivery are services built on top. The ConsensusEngine is built over the gossip KV, not the other way around. An agent that never calls consistent_set pays zero overhead for its existence.

Per-operation guarantee selection — same cluster, same binary, no separate infrastructure:
Operation Guarantee Cost
emit(signal) Best-effort, epidemic sub-ms — zero coordination
append("events/orders", bytes) Causally ordered, durable HLC stamp only — no broker
consistent_set("config/x", val) Linearizable consensus round-trip for this call only
distributed_lock("migration") Mutual exclusion consensus for this call only
Consul, Kafka, and Akka each pick one position on the consistency/availability tradeoff and apply it uniformly across your entire system. Mycelium picks per operation.

Layer 2 — Signal / Boundary Mesh

Complete · 213 tests

The Three Scopes

System
admits() → always true

All agents in the cluster act. Shed under load by opacity — use for broadcasts that can tolerate probabilistic delivery. Global events, cluster-wide state changes.

Group
admits() → groups.contains(g)

Only agents whose boundary includes the group name act. NLP agents, data processors, ops monitors — any named cohort. Shed under load by opacity. The primary routing primitive for work dispatch.

Individual
admits() → id == self.node_id

Only the target node acts. Direct replies (invoke.result) are the canonical use. Bypasses opacity — there is no routing alternative for a directed reply. Not subject to probabilistic shedding.

Complete Layer 2 API

// ── Group membership ──────────────────────────────────────────────────────
agent.join_group("nlp");          // gossips grp/nlp/<node_id> into KV store
agent.leave_group("nlp");
let groups: Vec<Arc<str>> = agent.groups();  // current boundary memberships

// ── Periodic heartbeat + pheromone trail ──────────────────────────────────
let _advert = agent.advertise(
    signal_kind::CONTRACT_AVAILABLE,
    SignalScope::Group("nlp".into()),
    Duration::from_secs(10),
    move || encode(&LoadState { .. }),
);
// advertise_persistent also writes svc/{kind}/{node_id} to Layer I on every tick.
// Late joiners discover via scan_prefix("svc/"); tombstoned on drop/shutdown.
let _advert = agent.advertise_persistent(kind, scope, interval, payload_fn);

// ── Emit / receive ────────────────────────────────────────────────────────
agent.emit("invoke", SignalScope::Group("nlp".into()), payload);
let result = agent.signal_once("invoke.result", Duration::from_secs(5),
    move |s| s.nonce == nonce).await;  // register BEFORE emitting

// ── Fault detection — supervisor pattern ─────────────────────────────────
// on_stale() called when last_signal(kind) is silent longer than threshold.
// Checks every threshold/4. Drop WatchHandle to cancel.
let _watcher = agent.watch(
    signal_kind::CONTRACT_AVAILABLE,
    Duration::from_secs(30),
    move || { respawn_worker(); },
);

// ── Quorum — threshold activation ────────────────────────────────────────
// Returns true when ≥ min_senders distinct NodeIds heard in window.
if agent.quorum(signal_kind::CLUSTER_EVENT, 3, Duration::from_secs(10)) {
    start_leader_election();
// quorum_persistent reads sys/quorum/ from Layer I — survives restarts.
let n = agent.quorum_persistent(signal_kind::CLUSTER_EVENT, Duration::from_secs(60));
}

// ── Inhibition — refractory period ───────────────────────────────────────
let mut invoke_rx = agent.signal_rx_with_capacity(signal_kind::INVOKE, 64);
tokio::spawn(async move {
    while let Some(sig) = invoke_rx.recv().await {
        // Block the next invocation for 500ms — prevents pile-up
        agent.suppress(signal_kind::INVOKE, Duration::from_millis(500));
        handle_invocation(sig).await;
    }
});

// ── Proactive opacity notification ────────────────────────────────────────
// Monitors fill_ratio; emits boundary.opaque / boundary.transparent to peers.
// Application suggests threshold; library adapts based on fill trend.
let _governor = agent.manage_opacity(
    signal_kind::BOUNDARY_OPAQUE,
    SignalScope::Group("nlp".into()),
    OpacityHint::default(),  // threshold=0.75, hysteresis=0.20
);

// Diagnostic scenario — worker pool appears stalled. Work through the layers:

// 1. Propagation health first — fix this before anything else
let stats = supervisor.system_stats();
if stats.dropped_frames > prev_dropped {
    tracing::warn!("gossip losing frames — raise max_concurrent_forwards");
}

// 2. Any worker heard recently?
let fresh = supervisor.last_signal(signal_kind::CONTRACT_AVAILABLE)
    .map(|t| t.elapsed() < Duration::from_secs(30))
    .unwrap_or(false);

// 3. Do we have quorum? (2+ distinct workers in 30s)
let pool_ready = supervisor.quorum(
    signal_kind::CONTRACT_AVAILABLE, 2, Duration::from_secs(30)
);

// 4. Authoritative state from pheromone trails
let live_trails = supervisor.scan_prefix("load/nlp/")
    .into_iter()
    .filter_map(|(_, b)| decode::<LoadState>(&b))
    .filter(|s| unix_ms_now() - s.written_at_ms < 30_000)
    .count();

// Trail present but quorum false → worker alive but signal channel suppressed or saturated
println!("signals fresh: {fresh}, quorum: {pool_ready}, live trails: {live_trails}");

Opacity vs Inhibition — Two Independent Mechanisms

Both mechanisms reduce signal delivery. They look superficially similar and are routinely confused. They are not — they arise from different causes and serve different purposes.

Opacity — passive, automatic, emergent
Triggered by
Handler channel fill ratio — automatic, no application code needed
Effect
Probabilistic shedding: admit_prob = 1 - fill_ratio. At fill=0.6, 60% of System/Group signals are shed.
Forwarding
Unaffected — the node still forwards every signal. The network stays fully connected.
Individual scope
Always bypassed — no routing alternative for a directed reply
Reversible
Auto — falls as the channel drains
Use for
Self-protection under load — emergent backpressure with no coordinator
Diagnose with
opacity(kind) → fill ratio 0.0–1.0
manage_ opacity
Does not change shedding itself. Adds a governor that watches fill ratio and tells peers via boundary.opaque when this node is becoming saturated. Library governs threshold; application provides a hint.
Inhibition — active, deterministic, application-controlled
Triggered by
Application calling suppress(kind, duration) — deliberate, programmatic
Effect
100% block — deterministic. Zero signals of that kind are delivered for the duration. Not probabilistic.
Forwarding
Unaffected — forwarding and last_signal timestamps continue normally
Individual scope
Blocked like any other scope — inhibition is unconditional
Reversible
Auto after duration; or explicit unsuppress(kind)
Use for
Refractory period, rate limiting, idempotency window, "just accepted work — stop accepting same kind"
Diagnose with
is_suppressed(kind) → bool
Biological analogy
The refractory period after a neuron fires — the cell explicitly will not fire again for a fixed window, regardless of stimulus.

Diagnostic shortcut: opacity(kind) > 0 means the node is overloaded. is_suppressed(kind) means the node chose to block it. Both can be true simultaneously — they are independent states.

Combined Scenario — Worker Under Heavy Load

// Three mechanisms working together — complementary, not redundant.

// 1. Opacity (automatic): as invoke_rx fills, boundary sheds incoming invocations
//    probabilistically. No code needed. Size the channel to match processing rate.
let mut invoke_rx = agent.signal_rx_with_capacity(signal_kind::INVOKE, 64);

// 2. Inhibition (active): after accepting one invocation, block the next for 500ms.
//    Prevents pile-up even when the channel is large enough to buffer many.
tokio::spawn(async move {
    while let Some(sig) = invoke_rx.recv().await {
        agent.suppress(signal_kind::INVOKE, Duration::from_millis(500));
        handle_invocation(sig).await;
    }
});

// 3. manage_opacity (proactive): emit boundary.opaque to peers when fill rises above
//    threshold — peers route new work elsewhere before the channel fully saturates.
let _governor = agent.manage_opacity(
    signal_kind::BOUNDARY_OPAQUE,
    SignalScope::Group("nlp".into()),
    OpacityHint::default(),
);

Capability Advertisement and Resolution

Nodes advertise what they offer and watch the matching set for changes. All state lives in the Layer I KV store under cap/{node_id}/{ns}/{name} — visible to any node at any time via scan_prefix("cap/") and anti-entropy-synced to late joiners.

// Advertise — reasserts cap/{node_id}/{ns}/{name} on an interval
let _handle = agent.advertise_capability(
    Capability::new("compute", "gpu"),
    Duration::from_secs(30),
);

// Resolve — snapshot of every node currently advertising a matching capability
let filter = CapFilter::new("compute", "gpu");
let matches: Vec<(NodeId, Capability)> = agent.resolve(&filter);

// Watch — push-based; debounced to a 50 ms idle window before firing
let mut rx = agent.watch_capabilities(filter);
rx.changed().await?;

Requirements and Demand Pressure

declare_requirement writes the node's declared need into req/. demand and watch_demand surface how many nodes want a capability versus how many provide it. demand_pressure = demanding / max(providers, 1). Pressure above 1.0 means demand outstrips supply. The library never auto-responds — this is a signal for orchestrators and autoscalers.

// Declare requirement — writes req/{node_id}/{ns}/{name} on an interval
let _h = agent.declare_requirement(CapFilter::new("compute", "gpu"), Duration::from_secs(30));

// Demand snapshot — pressure = demanding / max(providers, 1)
let s: DemandStatus = agent.demand(&CapFilter::new("compute", "gpu"));
println!("{} demanding, {} providing, pressure {:.2}",
    s.demanding_nodes.len(), s.providers.len(), s.demand_pressure);

// Push-based demand watch — fires on req/, cap/, or gcap/ changes matching filter
let mut rx = agent.watch_demand(CapFilter::new("compute", "gpu"));
rx.changed().await?;
if rx.borrow().demand_pressure > 2.0 { scale_up_gpu_workers(); }

Emergent Capability Groups

Nodes that share a capability self-assemble into a named group. The library projects their collective capability under gcap/{group}/{ns}/{name}/{contributor} and handles group-level requirement wiring. One consolidated task per group (not per member) keeps task count O(active groups).

// Any node advertising compute/gpu joins "gpu-pool" automatically.
// The library maintains gcap/ projections and group-level wiring.
agent.define_capability_group(
    "gpu-pool",
    CapabilityGroupDef {
        filter:   CapFilter::new("compute", "gpu"),
        provides: vec![Capability::new("compute", "gpu")],
        requires: vec![],
    },
    Duration::from_secs(60),
);

Inter-Group Wiring and Locality-Aware Routing

Wiring connects a consumer to provider groups without naming individual nodes. resolve_with_locality orders providers by shared topology prefix depth — same rack before same AZ before cross-AZ — and signal_wired_via_locality combines wiring and locality in one call.

// Signal via wiring — dispatches to all matching providers (direct + group members)
let outcome = agent.signal_wired_via(
    &CapFilter::new("compute", "gpu"), "render-job", payload,
).await;

// resolve_with_locality — (NodeId, Capability, depth) sorted by depth desc
// depth = length of shared locality prefix (e.g. 2 = same rack, 1 = same AZ)
let candidates = agent.resolve_with_locality(
    &CapFilter::new("render", "job"),
    LocalityPreference::PreferShared(0),  // prefer closest; fall back to any
);

// Locality path set in config — coarse to fine
config.locality_path = vec!["az1".into(), "rack2".into(), "host3".into()];

Consistency & Ordering — Opt-In Overlay

Planned

Traditional distributed frameworks pick one position on the CAP triangle and apply it uniformly. Consul/etcd pay Raft latency on every read/write. Kafka pays broker coordination on every publish. Akka pays ack overhead on every message. This architecture inverts that: the epidemic substrate is the foundation — always available, always fast — and you escalate to stronger guarantees only for the specific operation that demands it.

Most agent-to-agent coordination doesn't need linearizability — it needs fast and available. The rare operations that do need linearizability call consistent_set and pay consensus latency only for that call. Same cluster, same binary, no separate infrastructure for each tier.

2 · Route a Request
"Find something that can do X"
  • Emit to Group scope; workers compete to respond
  • Routing is emergent from opacity — no invoker selects a worker
  • Pheromone trails give the load picture; quorum confirms readiness
  • All via emit_async(kind, Group(scope), payload)
3 · Return a Correlated Result
"Here is the answer"
  • Actor reply, RPC response, MCP tool result — structurally identical
  • signal_once("result", timeout, |s| s.nonce == req_nonce)
  • Individual scope — bypasses opacity, always arrives
  • Substrate carries opaque Bytes; paradigm is above it

The reactive capability cell: each agent advertises what it can do, holds its authoritative state in KV (readable by anyone at any time), activates on threshold signal, writes results back to KV, and emits a correlated result signal. That model is a superset of all three paradigms — and it is the natural shape of the gossip substrate. The higher layers build on this shape; they do not replace it.

Actor / Event (Akka-style)

Actors are gossip agents with a defined key namespace (their state in KV) and a signal handler (their message loop). Location transparency comes from NodeId-based addressing — callers emit to Individual(node_id) without knowing the actor's network location. Supervision uses watch(): a supervisor watches a worker's heartbeat kind and triggers restart when stale.

// Actor registration
agent.set(format!("actors/summariser/{}", agent.node_id()), encode(&metadata));

// Message send (caller) — send to specific actor by NodeId
agent.emit("actor.msg", SignalScope::Individual(target_node_id), payload);

// Supervisor — restart if heartbeat goes silent for 30s
let _supervisor = agent.watch(
    signal_kind::CONTRACT_AVAILABLE,
    Duration::from_secs(30),
    move || { respawn(target_node_id); },
);

Async Services / RPC

RPC over gossip: invoker registers signal_once before emitting; worker handles via signal_rx and responds with Individual scope. Load balancing is emergent from opacity. Circuit breaking is last_signal staleness. Each service chooses its own serialisation — bincode for internal Rust services, JSON for MCP-compatible interfaces, protobuf for external-facing endpoints. The substrate routes by kind string; payload format is transparent to the gossip layer.

MCP — Server and Client Roles

The MCP bridge has two roles. Server role: Mycelium capability providers expose themselves as MCP tools — any external MCP client (Claude, a Python agent, a CLI) can discover and call them. Client role: Mycelium agents call external MCP tool servers; the bridge holds the outbound connection and re-enters results into the mesh as capability interactions. Credentials (API keys, OAuth) are held by the bridge layer, not the substrate. LangGraph is explicitly not a target — use MCP as the boundary instead.

// MCP server role — register a tool in the KV store
agent.advertise_capability(Capability::new("nlp", "summarise"), interval);
agent.set(format!("tools/summarise/{}", agent.node_id()), json_schema_bytes);

// MCP client discovery — any node sees all registered tools
let tools = agent.scan_prefix("tools/");

// MCP tool invocation — Layer 3 rpc_call formalises the signal_once + nonce pattern
let result = agent.rpc_call(tool_node_id, "mcp.invoke", json_rpc_bytes, timeout).await?;

// Python agent calling via the language bridge gateway
// mycelium.emit("nlp.summarise", scope="capability:nlp/summarise", payload=text)

Serialisation — Chosen at Need, Not Mandated

A new agent can join the mesh speaking a different serialisation format than its neighbours, advertise its capabilities via the shared kind namespace, and interoperate without requiring any existing agents to change. The gossip layer carries bytes; the capability advertisement carries the contract; agents that understand each other's payloads collaborate, those that don't simply don't handle that signal kind. Evolution happens by addition, not by coordinated migration.

Skills = Capabilities — no new primitive needed

The industry term "skill" maps directly onto advertise_capability with a richer JSON Schema attachment in KV. A skill is a named, discoverable, invocable unit of behavior — exactly what a capability already is. No separate skill registry, no skill execution engine, no new API surface.

A Skill Definition File (.skill.toml) declares the full contract for registering a capability and driving an LLM node:

# code_review.skill.toml
[capability]
ns          = "dev"
name        = "code-review"
description = "Reviews a PR diff and returns structured feedback"
ttl_secs    = 300

[capability.input]
type = "object"
required = ["pr_number"]
[capability.input.properties]
pr_number = { type = "integer" }
focus     = { type = "string", enum = ["security", "performance", "all"] }

[capability.output]
type = "object"
[capability.output.properties]
summary = { type = "string" }
issues  = { type = "array", items = { type = "string" } }
verdict = { type = "string", enum = ["approve", "request-changes", "comment"] }

[capability.policy]
max_concurrent     = 2
authorized_callers = ["orchestrator", "planner"]   # capability authorization scoping

[capability.platform]
requires = []       # e.g. ["gpu", "locality/east-0"] for platform-constrained skills

[skill]
prompt = """
You are reviewing a pull request. Given the PR number and focus area:
1. Fetch the diff via the gh tool
2. Analyse for the specified focus area
3. Return structured JSON matching the output schema
"""
tools = ["gh", "read_file"]   # mesh capabilities this skill may resolve and invoke

A SkillRunner node loads the file at startup, advertises the capability with schema in KV, and runs a signal_rx loop. On each invocation: deserialise input → run LLM with skill prompt → serialise output → respond via nonce RPC. One binary, any skill. Swap the file, get different mesh-visible behavior.

Skill composition is emergent. The tools list is resolved via resolve_capability at call time — not hardcoded at authoring time. Skills can invoke skills; the mesh routes the sub-invocation; the audit trail captures the causal chain. LLM credentials are held by the runner, not the substrate.

Plans — Not a Primitive

The industry term "plan" is not a Mycelium primitive. Planning is LLM-internal reasoning. Mycelium provides the execution substrate that makes any plan executable. Storing a plan as a first-class mesh object reintroduces a central scheduler — the same architectural problem as LangGraph.

// The "plan" lives in the LLM's context window, not the mesh
// LLM reasons → decides to invoke "dev/code-review"
let node = agent.resolve_capability(&CapFilter::new("dev", "code-review"));
let result = agent.signal_wired_via(&filter, "skill.invoke", json_payload).await;
// → result arrives via Individual scope nonce
// → LLM reasons again with result added to context
// → audit trail records the full causal chain keyed by HLC

The plan is the LLM's internal monologue. The audit trail is the post-hoc record. The mesh is the execution engine for both.

Layer 4 Security Primitives

Required alongside MCP and SkillRunner work — not a follow-up. Multi-agent MCP environments create threat models origin-based security (SOP, CORS) never covered. All three primitives address this at the mesh layer, not the transport layer.

PrimitiveWhat it doesMycelium hook

Capability authorization scoping Per-caller/session gate at resolve_capability. Only return capabilities the calling agent's context is authorized to see. Closes the confused deputy gap. [capability.policy].authorized_callers in skill/manifest; affects advertise_capability + resolve_capability signatures
Session-scoped mesh views LLM agent executing a task sees only capabilities authorized for that task's context — not the full capability space. Prevents cross-session leakage.

Token bloat and security scoping are the same design problem. A naive scan_prefix("tools/") dumps every capability schema into the LLM's context window — burning tokens on irrelevant tools and widening the confused deputy surface simultaneously. The fix is identical for both: tool discovery returns a filtered resolve_capability scoped to the caller's authorized context. One implementation, two benefits. Do not implement language bridge tool discovery as a raw scan and patch scoping in later.		 Scoping declaration constrains resolve_capability output per caller context; built on existing capability TTL. Language bridge tool-discovery endpoint must accept caller context from day one — not retrofittable.

Why mesh-level, not transport-level: Origin isolation, OAuth, and user confirmation are application-layer concerns. The confused deputy problem is about what an LLM decides to do within its legitimate access — that requires a capability gate, not a network boundary.

// ── Worker node ───────────────────────────────────────────────────────────
let agent = Arc::new(GossipAgent::new(node_id, config));
agent.start().await?;
agent.join_group("nlp");

// Advertise every 10s — emits signal (fast) + writes pheromone trail (durable)
let load_key = format!("load/{}", agent.node_id());
let _advert = agent.advertise(
    signal_kind::CONTRACT_AVAILABLE,
    SignalScope::Group("nlp".into()),
    Duration::from_secs(10),
    move || {
        let state = LoadState { queue_depth: QUEUE.len(), written_at_ms: unix_ms_now() };
        agent2.set(load_key.clone(), encode(&state));
        encode(&state)
    },
);

// Handle invocations with refractory period — prevents pile-up
let mut invoke_rx = agent.signal_rx_with_capacity(signal_kind::INVOKE, 64);
tokio::spawn(async move {
    while let Some(sig) = invoke_rx.recv().await {
        agent.suppress(signal_kind::INVOKE, Duration::from_millis(500));
        let req: InvokeRequest = decode(&sig.payload);
        let result = run_model(&req.payload).await;
        agent.emit(
            signal_kind::INVOKE_RESULT,
            SignalScope::Individual(sig.sender),
            encode(InvokeResponse { nonce: req.nonce, result }),
        );
    }
});

// Proactive peer notification when load builds
let _governor = agent.manage_opacity(
    signal_kind::BOUNDARY_OPAQUE,
    SignalScope::Group("nlp".into()),
    OpacityHint::default(),
);

// ── Invoker node ──────────────────────────────────────────────────────────
let nonce = fastrand::u64(..);
let reply_fut = agent.signal_once(  // register BEFORE emitting
    signal_kind::INVOKE_RESULT,
    Duration::from_secs(5),
    move |s| s.nonce == nonce,
);
agent.emit_async(signal_kind::INVOKE, SignalScope::Group("nlp".into()), payload).await;

match reply_fut.await {
    Some(sig) => decode(&sig.payload),
    None => {
        // No reply — check pheromone trails to decide retry vs. escalate
        let any_live = agent.scan_prefix("load/").into_iter()
            .filter_map(|(_, b)| decode::<LoadState>(&b))
            .any(|s| unix_ms_now() - s.written_at_ms < 30_000);
        if any_live { retry_with_backoff() } else { Err("no workers") }
    }
}

// ── Supervisor node ───────────────────────────────────────────────────────
// Watch the worker's heartbeat — call respawn() if silent for 30s
let _watcher = supervisor.watch(
    signal_kind::CONTRACT_AVAILABLE,
    Duration::from_secs(30),
    move || { respawn_worker(); },
);

// React to peer overload signal — adjust routing immediately
let mut opaque_rx = supervisor.signal_rx(signal_kind::BOUNDARY_OPAQUE);
tokio::spawn(async move {
    while let Some(sig) = opaque_rx.recv().await {
        route_around(sig.sender);  // fast-path: before pheromone trail updates
    }
});

Future Research Directions

Horizon

Not on the near-term roadmap — but the natural next layer of the chemical signalling metaphor. These directions are worth naming because they inform architectural decisions made now.

🕸
Fitness-Weighted Topology

Activity weighting (prefer recently-active peers) is a step. The natural next layer is connections weighted by usefulness — how predictive is peer X's state for my decisions? A connection-strength field on the peer table, updated by the application layer and consulted by the forwarding logic, would make the topology self-organise around actual traffic patterns. The KV store and subscriptions already provide the ingredients.

∇
Gradient / Accumulation Signals

The pheromone trail pattern hints at this, but last-write-wins is a poor substrate for accumulation. Gradient signals that accumulate across hops — summing, averaging, max-pooling — rather than last-writing-winning would make the epidemic layer directly expressive for neural-style activation propagation. This is a deeper change: accumulation semantics conflict with LWW idempotency. But it is the natural next layer for the chemical computing metaphor.

🔑
Key Ownership & Migration

The current convention — each agent writes only its own keys — is unenforced. For an evolvable substrate where agents can be added, removed, or repurposed at runtime, informal ownership becomes a problem, especially if agent identity and key namespace are decoupled during evolution. The current model is correct as long as the agent-to-cell mapping is stable. Some notion of ownership transfer or key migration may eventually be needed.

1 · Broker-less scope filtering
with epidemic delivery guarantees
  • NATS, Kafka require a broker cluster to route by scope
  • Here scope is a pure application-layer filter on an epidemic substrate
  • No routing infrastructure to operate or fail
  • Group signals reach every node; boundary decides action
2 · Group topology as KV state
in the same store, not a separate registry
  • Most systems: separate Consul/etcd + separate message bus
  • Here group membership IS a gossip KV entry
  • Same propagation, same LWW, same visibility
  • Cluster is self-describing — zero extra infrastructure
3 · State and events on one transport
one wire format, two semantic modes
  • KV state: LWW, persistent, queryable, anti-entropy sync'd
  • Signals: TTL-bounded, ephemeral, nonce-dedup'd
  • Typically requires two separate systems
  • One embedded library; no external process to operate
4 · Serialisation autonomy
evolution by addition, not migration
  • A new agent joins speaking a different format — no cluster migration needed
  • Capability advertisement uses kind strings; payload format is opaque
  • MCP bridge: JSON. Internal agents: bincode. External: protobuf. Coexist.
  • No cluster-wide serialisation contract to maintain
5 · NodeId as the only contract address
no HTTP endpoints, no service registry
  • The gossip identity is the address — nothing else to manage
  • No HTTP endpoint to provision, rotate, or health-check

  • Location transparency falls out of the substrate — not bolted on

The core philosophy — epidemic state, threshold-activated events, evolving topology, no central coordinator — is the right shape for adaptive AI systems. The chemical signalling metaphor isn't decorative; it's doing real architectural work. The particular assembly of epidemic KV state, ephemeral scoped signals, stigmergic pheromone trails, emergent load-adaptive routing, and now active inhibitory signals, in a single embedded library grounded in biological CAS principles as first-class design constraints, has no direct precedent in the distributed systems literature. The two-timescale coordination model (neural signals + hormonal trails) applied to distributed AI agent orchestration is the differentiated position.

The substrate is architecturally novel and coherent. The ideas are well-executed and genuinely more sophisticated than anything in the current AI agent framework landscape at the gossip/capability layer. 4 — Security configuration
Complete — 2026-05-24

Enable the tls cargo feature and set GossipConfig::tls. TLS is opt-in; omitting the field leaves all existing plain-TCP behaviour unchanged.

Quick start (auto-generate):

[dependencies]
mycelium = { version = "…", features = ["tls"] }
// Auto-generates CA + per-node Ed25519 cert on first start.
// Certs are saved in ./mycelium-tls/ and reused on restart.
config.tls = Some(TlsConfig::default());

Manual PEM paths:

config.tls = Some(TlsConfig {
    cert_pem:     Some("/etc/mycelium/node.crt".into()),
    key_pem:      Some("/etc/mycelium/node.key".into()),
    ca_cert_pem:  Some("/etc/mycelium/ca.crt".into()),
    ..TlsConfig::default()
});
  • Auto-generate flow: on first start each node writes auto_cert_dir/{node_id}.key (raw 32-byte Ed25519 key) and shares a cluster CA. Distribute ca-cert.pem to all nodes before starting the cluster; each node re-generates its own cert from the shared CA on every start.
  • Identity publication: the 32-byte Ed25519 verifying key is gossiped to sys/identity/{node} so peers can verify signed consensus messages even before a direct mTLS handshake has occurred.
  • Consensus signing: all Propose, Vote, Nack, and Commit payloads are Ed25519-signed by the sender and verified on receipt. Forged ballots are silently dropped.
  • Mixed-mode: a node without a matching cluster CA is rejected at the TCP handshake. Plain (no-TLS) nodes and TLS nodes cannot interoperate — this is intentional.
  • Network policy: only the gossip TCP port needs to be open between cluster nodes; the mTLS handshake is the access control boundary. No other ports are required.
  • Not yet implemented: hot certificate rotation without cluster disruption; KV write signing (each gossip hop would need to re-sign — deferred).
