When the Cluster Is the Queue
Mycelium Project · 2026

A new primitive for distributed AI pipelines where the buffer, the scheduler, and the routing table are all emergent properties of a single gossip substrate — the same one that's already running as part of your cluster. No brokers. No dead-letter queues. No static topology.

§1

The Unix Pipe Moment

In 1973, Doug McIlroy gave the world the pipe: cat file | grep pattern | sort. Programs that knew nothing about each other could compose. The medium was the protocol. stdout of one process became stdin of the next. No orchestrator. No registry. No broker. Just a channel.

That idea — that the channel itself is the coordination primitive — is one of the most powerful ideas in computing. It scaled from shell scripts to programming languages (functional pipelines, lazy iterators, Rx streams). But something happened when we tried to bring it to distributed systems.

We broke the channel into pieces. We put a message broker between producer and consumer. We added a task queue. We added a worker registry. We added a scheduler. We gave each of those pieces its own failure mode, its own ops surface, its own monitoring stack. The simplicity of the pipe disappeared into operational complexity.

"The pipe is so simple it feels trivial. The distributed equivalent is so complex it requires a dedicated platform team."

This document is about a pattern that recovers the pipe's simplicity at distributed scale — without sacrificing correctness or resilience. We call it an Agentic Flow Network.

§2

The Separation Tax

Consider a typical ML pipeline: raw data → ingestion → feature extraction → model inference → post-processing → storage. In 2024, the standard deployment looks like this:

Infrastructureconventional ML pipeline topology
Workers     → Redis          (work queue)
Workers     → Consul         (service registry)
Workers     → Celery         (task scheduler + retries)
Workers     → Kafka          (event bus between stages)
Workers     → Dead-letter Q  (failure handling)
Workers     → Prometheus     (queue depth monitoring)

6 extra systems · 6 failure modes · 6 ops surfaces

Every one of those systems was added to solve a real problem. Redis because workers need to share work. Consul because workers need to find each other. Celery because work needs retries and timeouts. Kafka because stages need to communicate asynchronously. The dead-letter queue because work sometimes fails. Prometheus because you need to see what's happening.

We call this the Separation Tax: every separation between compute and state is a new system to deploy, monitor, and fail. The tax compounds. Each system added to solve the previous one's shortcomings creates new shortcomings of its own.

The tax is not about technology choices. Redis is excellent. Kafka is excellent. The tax is about the structural assumption that compute and state must be separate systems. Once you accept that assumption, the complexity is load-bearing.

§3

What Flow-Based Programming Got Right

In the early 1970s, J. Paul Morrison at IBM developed Flow-Based Programming (FBP): networks of independent black-box processes that communicate by passing information packets through bounded channels. Processes know nothing about each other. The topology — which process connects to which — is the program.

FBP got several things right that the industry keeps rediscovering:

  • Processes should be reusable and independently testable
  • Backpressure should be automatic (bounded channels block producers)
  • The topology should be declarative, not embedded in process code
  • Dead processes should not block the whole network

These principles are correct. But FBP implementations share a critical limitation: the channels are external to the compute nodes. Workers must connect to a runtime that manages channels. The separation tax exists in FBP too — it's just called "the FBP runtime" instead of "Kafka."

The question that Agentic Flow Networks answer is: what happens if the channel is the compute node?

§4

The Substrate Unity Insight

Modern gossip-based distributed systems — Mycelium, CRDTs, epidemic KV stores — give every node a local replica of the cluster's shared state. A write on node A propagates to all other nodes within milliseconds. A read on any node is a local memory access.

This is usually described as a consistency mechanism. But it has an underexplored consequence: every worker already holds the buffer.

"When a worker writes its output to the KV ring, that value is immediately available for local reads on every other node. There is no queue service to query. The scan latency is zero."

Let that sink in. The KV ring is the buffer. It's already there — running as part of the cluster, replicated to every node, consistent within milliseconds. You don't need to deploy a queue service to get a distributed work queue. You already have one.

The second half of the insight: capability gossip as the scheduler. Each worker advertises its capabilities — what stage methods it can handle — by writing capability records into the same KV substrate. A coordinator calls resolve_capability("stage_c", "worker") to find nodes that can run stage C. The response is a list of live nodes, resolved locally, without a network hop to a service registry.

This is Substrate Unity: the buffer, the scheduler, and the routing table are all emergent properties of the same gossip substrate. There is no separation — and therefore no Separation Tax.

§4b

From Push to Pull — the TupleSpace Refinement

The KV-ring-as-buffer (§4) is the right first move, but it leaves one coordinator-shaped seam: a coordinator still decides which worker gets each item. It calls resolve_capability, picks a node, and pushes. To pick well it needs a model of every worker's load — and that model goes stale between the read and the dispatch. Staleness is precisely what drives misrouting and the audit burden the whole architecture set out to eliminate. The buffer was freed from the Separation Tax; the decision was not.

The refinement is to remove the decision-maker entirely by inverting the direction of control: instead of a coordinator pushing work toward a predicted worker, idle workers pull work when they are ready. Readiness is announced by the claim itself, so there is no model to keep fresh — the variable staleness would corrupt does not exist. This is Gelernter's tuple space (Linda, 1985), and the classical objection to it — that the shared space is itself a coordinator needing its own consistency machinery — dissolves when the space is built on a coordinator-free substrate rather than beside one.

Shipped as constructive proof. The mycelium-tuple-space companion crate rebuilds work distribution — the most coordinator-shaped workload there is — using only the substrate's public API, with no component that predicts any agent's state. Producers put items into named stages; workers block on take and receive work exactly when ready. The serving node is discovered by capability advertisement, its failure is detected by TTL evaporation, and a mirror promotes itself through the same emergent election as any other role. 1,000 items across ten producers and ten workers are delivered exactly once with no participant ever holding a view of another's load; kill the serving node mid-flight and a standby serves the survivors without resurrecting acknowledged work. The pull pattern survives; only the coordinator-shaped infrastructure around it was ever wrong.

The §5 properties below describe the AFN concept in its general form; the TupleSpace companion is its purest instantiation — flow-based programming with the last decision-maker removed.

§5

Five Properties of an Agentic Flow Network

A system is an Agentic Flow Network if and only if it exhibits all five of these properties. They are not aspirational — they are structural requirements.

I
Substrate Unity
The buffer, scheduler, and routing table are all emergent properties of a single gossip substrate. No separate queue, registry, or router.
II
Topology Emergence
Routing topology is not configured. It emerges from capability gossip. Add a node — it joins. Remove a node — its TTL expires. The topology is always live and always correct.
III
Fluid Allocation
Workers have no fixed stage assignment. They follow the work. When Stage A drains, those workers immediately become available for Stage B — without reconfiguration.
IV
Opacity Backpressure
Saturated workers signal unavailability, becoming invisible to routing. Backpressure is automatic, measured, and self-correcting. No explicit rate limiting required.
V
TTL-Native Cleanup
Abandoned work items expire automatically. No dead-letter queues. No manual intervention for crashed workers. The cluster heals itself through TTL expiry and anti-entropy.
§6

The Pattern in Practice — Stage A → B → C → D

Here is the fluid pipeline pattern made concrete. Ten workers. 200 news articles. Four stages: Parse → Enrich → Score (simulates LLM) → Aggregate.

Step 1: Seed the buffer. The coordinator writes all 200 work items into the KV ring. Each write propagates to all 10 workers via gossip. Within seconds, every node holds a replica of the full stage-A buffer.

Pythoncoordinator — seeding the KV buffer
# 200 writes, each propagates epidemically to all 10 workers
for article in articles:
    agent.set(f"pipeline/stage-a/{article['id']}",
             json.dumps(article).encode())

# scan_prefix is a LOCAL read — zero network latency
items = agent.keys(prefix="pipeline/stage-a/")
# → ['pipeline/stage-a/article-0000', 'pipeline/stage-a/article-0001', …]

Step 2: Workers advertise all capabilities. Each worker advertises itself as a handler for all four stage methods at startup. No static assignment — the same worker will handle Parse requests today and Score requests tomorrow, or both in the same run.

Pythonworker — fluid capability advertisement
# Every worker advertises ALL four stage capabilities.
# The coordinator routes to whichever stage has work.
for ns in ("stage_a", "stage_b", "stage_c", "stage_d"):
    agent.advertise_capability(ns, "worker", interval_secs=15)

# Serve all four methods concurrently — no reconfiguration ever needed
await asyncio.gather(
    serve_stage(agent, "stage_a.parse",     parse_article),
    serve_stage(agent, "stage_b.enrich",    enrich_article),
    serve_stage(agent, "stage_c.score",     score_article),
    serve_stage(agent, "stage_d.aggregate", aggregate_article),
)

Step 3: The coordinator drains each stage. The drain_stage loop scans the current stage's KV prefix, resolves free workers via capability query, and dispatches RPC calls. When workers complete, they write results to the next stage's KV prefix and delete from the current one. The loop exits when the prefix is empty and no tasks are in flight.

The wave: As Stage A drains into B, all 10 workers shift from stage_a.parse calls to stage_b.enrich calls — without any instruction from the coordinator. The capability ring reflects what work is available; workers respond to what the coordinator routes. The wave is emergent.

§8

When to Use It — and When Not To

AFN is the right pattern when:

  • Work items are independent (no inter-item ordering requirement)
  • Stages have varying latency — a slow stage benefits from the entire pool concentrating there
  • The worker pool is elastic — nodes join and leave during operation
  • You are on the edge or air-gapped — no external broker infrastructure available
  • You want operational simplicity — one system, one failure mode, one monitoring surface
  • Work items can be abandoned and retried — at-least-once via TTL re-queue, not strict exactly-once

AFN is the wrong pattern when:

  • Strict ordering is required — the KV LWW model does not preserve enqueue order
  • Exactly-once delivery semantics are mandatory — use Kafka with its transactional API
  • Work items have complex interdependencies — use a DAG scheduler (Airflow, Dagster)
  • Long-term event log retention is needed — use a durable append-only log
  • You need cross-cluster, cross-datacenter fan-out — gossip TTL windows may not align

The sweet spot: AI inference pipelines with variable-latency stages and a fixed-size worker pool. Exactly the workload that motivated this pattern — where LLM inference at Stage C creates a bottleneck that benefits from having all available workers route toward it automatically.

§10

Replacing the Simulated LLM with a Real One — Prompt Skills

Stage C in the demo sleeps for STAGE_C_SLEEP seconds to stand in for a real model inference call. The llm cargo feature gives you the real thing: prompt templates stored in the gossip KV ring, served by whichever node has a backend registered, and called through the same capability ring as every other stage.

Swap score.py's simulated sleep for a live model call in three steps:

Step 1 — Register the skill on a hosting node (Rust). One node (or several, for redundancy) calls register_prompt_skill. The template is written to the cluster KV store with a one-week TTL and propagates to every worker via gossip. The model name is baked into the OpenAiBackend at construction — the template carries no model field, keeping node-local knowledge local.

Rustskill host — one node registers the scoring template
use mycelium::{OpenAiBackend, PromptTemplate, GossipAgent};

let backend = OpenAiBackend::new(
    "http://localhost:11434/v1",  // any OpenAI-compatible endpoint
    "",                            // API key (empty for local Ollama)
    "llama3.2",                    // model — baked in here, not in the template
);
let template = PromptTemplate {
    system: "You score news articles. Return JSON only: \
             {\"relevance\":0.0,\"urgency\":0.0,\"credibility\":0.0,\"composite\":0.0}".into(),
    user_template: "Title: {{input}}\nScore this article on relevance (0-1), \
                    urgency (0-1), credibility (0-1).".into(),
    max_tokens: 128,
    temperature: 0.2,
    metadata: Default::default(),
};
// Template written to KV, capability advertised — all workers can resolve it.
let _skill = agent.register_prompt_skill("stage_c", "score", template, backend).await?;

Step 2 — Call the skill from Stage C (Python worker). Replace time.sleep(STAGE_C_SLEEP) in stages/score.py with a gateway call to /gateway/llm/call. The Python SDK wraps this in one line.

Pythonstages/score.py — replace simulated sleep with live inference
from mycelium.prompt_skill import PromptSkillClient

_client = PromptSkillClient("http://localhost:8300")

def score_article(payload: dict) -> dict:
    # Previously: time.sleep(STAGE_C_SLEEP)
    result = _client.call("stage_c", "score",
                          input=payload["title"], timeout_secs=30.0)

    # result.output is the JSON string the model returned
    scores = json.loads(result.output)
    return {**payload, "scores": scores, "model_used": result.model_used}

Step 3 — Nothing else changes. The coordinator's drain_stage loop, the KV buffer, the fluid allocation, and the TTL cleanup all work exactly as before. Stage C is now slower (real inference instead of a sleep), so more workers pile up there — and the fluid pool concentrates resources there automatically, exactly as the AFN pattern predicts.

Template update with no restart. While the pipeline is running, call PUT /gateway/prompts/stage_c/score on any node with an updated template body. The write propagates via gossip KV to every worker. The next invocation — on any node — picks up the new template from the local KV snapshot. No rolling restart. No coordination. The substrate handles it.

Shelllive template update — no restart
curl -X PUT http://localhost:8300/gateway/prompts/stage_c/score \
     -H 'Content-Type: application/json' \
     -d '{"system":"Updated scoring prompt.","user_template":"{{input}}","max_tokens":128,"temperature":0.1}'
# propagates to all workers via gossip KV; next invocation uses new template

Standalone example — no pipeline needed:
cargo run --example prompt_skill_demo --features llm
